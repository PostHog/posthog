# Quill Tokens — Agent Guide

`@posthog/quill-tokens` is the source of truth for quill's design tokens. Tokens
are authored in TypeScript under `src/` and **generated** into CSS in `dist/`.
Never hand-edit `dist/*.css` — every file is stamped
`Auto-generated by @posthog/quill-tokens — do not edit manually`.

## Pipeline

```text
src/*.ts  ──(tsx src/build.ts)──▶  dist/*.css
```

`pnpm --filter @posthog/quill-tokens build` runs `src/build.ts` then `vite build`.
`build.ts` calls the generators in `colors.ts` and writes:

| File               | Contents                                                     | Consumed by                            |
| ------------------ | ------------------------------------------------------------ | -------------------------------------- |
| `color-system.css` | `:root` (light) + dark-selector overrides — runtime CSS vars | apps & packages at runtime             |
| `tailwind.css`     | `@theme` + `@custom-variant` + base layer                    | apps                                   |
| `tailwind-lib.css` | `@theme` + `@custom-variant` only                            | library packages                       |
| `*.scoped.css`     | same, gated behind `[data-quill]`                            | consumers with their own design system |

After changing anything in `src/`, **rerun the build** so `dist/` matches.
(The quill storybook also watches `src/` via `quillTokensWatcher` in its
`main.ts`, so it hot-reloads without a manual build.)

## Token groups

- **Semantic colors** (`colors.ts`, `buildSemanticColors`) — `[light, dark, tailwindClass]`
  tuples. Surface/neutral colors are theme-hue-derived (`surface()`); status and
  brand colors are static `oklch()`. Theme-derived tokens must be listed in
  `THEME_DERIVED_TOKENS` (a load-time guard enforces this) so they emit on `*`
  and re-evaluate under local `[--theme-hue:X]` overrides.
- **Data-visualization** (`data-viz.ts`) — see below.
- **Spacing / typography / shadow / border-radius** — own modules, own generators.

## Data-visualization tokens (`data-viz.ts`)

Consumed by `@posthog/quill-charts`, which ships **no CSS** and reads these vars
into a `ChartTheme` at runtime (via its `themeFromCssVars` / `useChartTheme`).

- `--data-color-1..15` — an **ordered categorical palette** for series colors.
  Order is meaningful and hues must stay mutually distinguishable. Values are
  static brand hexes (not theme-hue-derived); only the entries needing extra
  dark-mode contrast carry an override. Edit the `dataColors` array — the no-CSS
  fallback copy `DEFAULT_CHART_COLORS` in quill-charts (`core/theme.ts`) must
  match; its `theme.test.ts` asserts equality with `dataColorPalette`, so drift
  fails CI rather than slipping through.
- `--color-graph-axis-label`, `--color-graph-axis-line`, `--color-graph-crosshair`
  — graph chrome, aliased onto semantic tokens (`--muted-foreground`, `--border`)
  so they flip with light/dark automatically (no dark block needed).

These names match the PostHog app's historical `--data-color-*` / `--color-graph-*`,
so the app can drop its local definitions and inherit quill's instead.

JS exports for non-CSS consumers: `dataColors`, `dataColorPalette`, `dataColorVarName`.

## Adding or changing a token

1. Edit the relevant `src/*.ts` module (e.g. add a `dataColors` entry, or a
   `semanticColors` tuple).
2. If a semantic token references a theme var directly, add it to
   `THEME_DERIVED_TOKENS`.
3. Rerun `pnpm --filter @posthog/quill-tokens build`.
4. Verify the var appears in `dist/color-system.css` (and `.scoped.css`).
