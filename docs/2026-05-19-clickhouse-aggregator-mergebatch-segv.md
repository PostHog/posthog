# ClickHouse SIGSEGV in `Aggregator::mergeBatch` on `*-offline` node (26.3.10.60)

## Context

`system.crash_log` on `prod-us-iad-ch-1e-offline.internal.ec2.us-east-1.prod-us.posthog.dev`
shows four SIGSEGVs on 2026-05-19 between 10:48 and 11:34 UTC, all with the same
signature. The questions to answer are: **what is crashing**, **is it our query
or ClickHouse**, and **what mitigation should we take**.

- An aggregate function holding **String state in the Arena** (`anyIf(String)`,

- The crash is in the **second-level merge** phase
  (`MergingAggregatedBucketTransform → mergeBlocks → mergeStreamsImpl → mergeBatch`),
  not in the per-block aggregation.

## Root cause

This is a **ClickHouse server bug**, not a malformed query on our side.

Evidence:

1. **`SEGV_MAPERR` inside `mergeBatch` for a String-state aggregate function** is the
   textbook signature of a use-after-free / wrong-arena-pointer in the
   two-level merge path. The merge function dereferences a `char*` that lives
   inside a `SingleValueDataString` (or `Merge`-combinator state); if the
   originating arena has been freed or the state pointer was never properly
   written, you read into unmapped memory.
2. **Row 3's fault address** is `27692494056993847` = `0x625F3838635F6437`. That
   is not a plausible pointer — it is ASCII-range bytes (`b_88c_d7` pattern).
   The aggregator interpreted **inline string data** as a pointer field of the
   state struct. That is corruption of the aggregate-state header during merge,
   not "we ran out of memory" or "OOM killer."
3. **Rows 1, 2, 4** all hit the exact same fault address (`0xFFFE_3D80_0000`-ish
   when read as little-endian). Identical addresses across unrelated queries
   point to a deterministic offset into a freed/decommitted arena page —
   consistent with arena lifetime bug, not random memory pressure.
4. All four crashes are on a **single host** and the **same CH build**. Other
   replicas are not crashing on identical queries, so the issue is build-/
   version-specific (likely a regression in 26.3.x's
   `MergingAggregatedBucketTransform` arena handling).
5. The two queries are not the same; the only thing they share is "GROUP BY
   String key, aggregate that owns String state, two-level merge."

The PostHog query is correct — it has been running for months on prior
ClickHouse versions without this signature. The trigger is the upgrade of this
node to **26.3.10.60**.

## Recommendation

Treat as a ClickHouse upgrade regression on the `*-offline` node. In priority
order:

1. **Roll back** the `prod-us-iad-ch-1e-offline` node to the previous CH
   version, or pin 26.3.10.60 only to a canary while we investigate.
2. **File upstream** at `github.com/ClickHouse/ClickHouse/issues` with:
   stack trace, fault addresses, version + git hash, both query texts, and a
   minimal repro. Search first for existing reports — keywords:
   `MergingAggregatedBucketTransform mergeBatch SEGV AggregationMethodString`.
   Likely already tracked for 26.3.
3. **Server-side workaround** if rollback is not acceptable, applied via
   `users.xml` profile on the affected node (no app change required):
   - `distributed_aggregation_memory_efficient = 0` — bypasses
     `MergingAggregatedBucketTransform`, which is the crashing transform.
   - or raise `group_by_two_level_threshold` / `group_by_two_level_threshold_bytes`
     high enough to skip two-level aggregation for these queries.
   Both are blunt — they will increase memory usage for large group-bys — but
   they sidestep the bucket-merge path.
4. **No PostHog code change is warranted** for the deletion query
   (`anyIf(distinct_id, notEmpty(distinct_id)) IN (...)`). It's idiomatic, has
   shipped for ages, and rewriting it to dodge a server bug obscures the real
   problem. Only revisit if (1) rollback is impossible **and** (2) the
   workaround in step 3 is unacceptable.

## Verification

1. Read recent CH changelog entries between the prior offline-node version and
   26.3.10.60 — look for fixes/regressions in `Aggregator`,
   `MergingAggregatedBucketTransform`, two-level aggregation, or
   `SingleValueDataString`.
2. Re-run both crashing queries (with the exact `team_id` and parameters from
   `system.crash_log`) against:
   - the offline node — expect crash to reproduce, possibly intermittently;
   - any non-offline replica on the prior version — expect no crash.
   This confirms the regression is build-specific.
3. After applying the chosen mitigation (rollback or
   `distributed_aggregation_memory_efficient=0`), watch `system.crash_log` on
   the affected host for 24 h:
   ```sql
   SELECT event_time, query_id, signal_description, fault_address
   FROM system.crash_log
   WHERE hostname = 'prod-us-iad-ch-1e-offline.internal.ec2.us-east-1.prod-us.posthog.dev'
     AND event_time > now() - INTERVAL 1 DAY
   ORDER BY event_time DESC;
   ```
   Expect zero new rows with this signature.
