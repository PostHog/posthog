# Cymbal crate discovery for agents

Use this as the first stop when you need to choose where to edit under `rust/cymbal/crates/`.
It is intentionally a navigation aid, not the architecture source of truth; see `../README.md`, `../docs/architecture.md`, `../docs/operations.md`, and `../docs/compatibility.md` for broader context and decisions.
Run commands from the repository root unless noted otherwise.

## How to choose the right crate

- Public gRPC/protobuf changes: start in `crates/api`, then update `crates/server/src/api.rs` for wire/domain conversion and regenerate/check generated code as needed.
- Internal stage transport: start in `crates/api/proto/cymbal/v1/stage.proto`, then `crates/server/src/stage.rs`, `crates/server/src/remote.rs`, and `crates/server/src/remote_runner.rs`.
- Reusable stage framework primitives, linear pipeline contract validation/runner APIs, generic circuit-breaker mechanics, generic rate-limit/admission primitives, or dependency-light routing/capacity/fallback logic: start in `crates/core`.
- Exception-event payload contracts (`InputEvent`, `EventResult`, `EventOutcome`, `RateLimitGateOutput`, `RateLimitDecision`, `RateLimitMode`) live in `crates/domain`; update their `StagePayload::TYPE` versions and `crates/server/src/registry.rs` only when remote payload shapes intentionally change.
- Exception-specific pipeline adapters, concrete error-tracking stage order, fan-in/fan-out, or final result ordering: edit `crates/pipeline`; keep transport and config in `crates/server`.
- Symbolication or symbol artifacts: use `crates/symbolication` for language-specific frame resolution and `crates/symbol-store` for fetch/cache/parse/save boundaries.
- Grouping and fingerprinting: use `crates/stages/grouping` for stage orchestration, `crates/fingerprinting` for automatic/manual fingerprint construction, and `crates/rules` for HogVM-backed rule evaluation.
- Issue persistence, suppression, assignment, and PostHog capture side effects: start in `crates/stages/linking`; put concrete Postgres/Redis/PostHog integrations in `crates/repositories` and wire them through `crates/runtime`.
- Runtime config and dependency construction: edit `crates/runtime`; use `crates/server/src/config.rs` only for server/transport knobs.
- Server orchestration, local/remote stage placement, transport, deployment config, limits, readiness, and shutdown: edit `crates/server`.
- Alerting side effects and spike detection: edit `crates/stages/alerting`; keep notification or persistence implementations behind dependencies wired by `crates/runtime`/`crates/repositories`.

## Workspace checks commonly paired with crate edits

- Metadata/discovery smoke: `cargo metadata --manifest-path rust/Cargo.toml --format-version 1 --no-deps`
- Formatting: `cargo fmt --check --manifest-path rust/Cargo.toml --all`
- Whitespace: `git diff --check`

If `cargo fmt --check --manifest-path rust/Cargo.toml` reports a virtual-manifest target issue, keep the documented `--all` form above or run the equivalent from `rust/`.
