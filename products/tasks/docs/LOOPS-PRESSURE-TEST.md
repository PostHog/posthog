# Loops plan — pressure test

Status: resolved review artifact, kept for history. These findings (plus a
second fact-check and gap-hunt round) are folded into `LOOPS.md` draft v1;
the verdict below refers to the superseded v0. Generated by an 8-lens parallel
pressure test of `LOOPS.md` (draft spec v0). Each lens ran independently
(concurrency/idempotency, data model & migrations, security & tenant isolation,
GitHub webhook infra & ops, cross-repo dependency sequencing, cost & abuse,
product/phasing/notifications, adversarial red-team). Findings below are
deduplicated and grouped; severities are as assigned by each reviewer.

## Verdict

🚫 **Not ready for Phase 1 engineering start.** Two classes of finding are
blocking, not polish:

1. A handful of concrete correctness gaps (overlap TOCTOU, dedup scoping,
   retention vs. active runs, `LoopTrigger` team-scoping) that will misbehave
   on day one, not under adversarial load.
2. One mechanism the doc _asserts_ exists but doesn't design: bounding
   phase-2 "PR babysitting" follow-up runs across workflow boundaries. As
   written, `max_fix_iterations` only lives inside a single Temporal
   workflow; nothing in the schema carries that counter across the
   webhook-spawned follow-up runs the doc says are "bounded by the same
   iteration cap." This is the single most dangerous unaddressed item — a
   plausible unbounded auto-fix loop against one PR that every stated
   guardrail (auto-pause, cooldown, usage gate) is blind to, because each hop
   reports as an individual success.

Everything else is real but fixable during implementation once the above are
resolved.

## 🔴 Critical / blocking

### 1. Phase-2 babysitting has no persisted iteration counter across run boundaries

**Section:** Behaviors ("Phase 2 of babysitting"), Run.
`max_fix_iterations` today is enforced as in-workflow state (`MAX_CI_REPETITIONS`)
inside one Temporal execution. Phase 2 spawns a **new** Task/TaskRun/workflow
per webhook fire on a loop-created PR — there is no field on `Loop`, `Task`,
or `TaskRun` (`fix_iteration_count`, `run_chain_id`, `origin_run_id`, ...) to
carry a count across that boundary. "Bounded by the same iteration cap,
recorded on the run chain" describes a mechanism the data model doesn't have
storage for.
**Concrete failure:** a human leaves one comment on a stale loop-created PR
weeks after the original run exhausted its budget and ended "successfully" →
new run spawned with iteration count reset to zero → fixes, fails, gets
re-triggered again → repeats indefinitely. Each hop is an individual
"success," so `consecutive_failures`/auto-pause never engages.
**Fix:** persist iteration count on the run chain explicitly (e.g.
`TaskRun.loop_chain_iteration` seeded from parent run + 1), checked against
`max_fix_iterations` _before_ spawning a phase-2 follow-up, not asserted in
prose.

### 2. Self-feeding `push`-trigger + auto-fix loop is a mechanical consequence, not a hypothetical

**Section:** Behaviors, GitHub event triggers §2, PR babysitting phase 2.
`fix_review_comments=true` pushes fix commits. A `push`-event `LoopTrigger`
on the same repo has no exclusion for commits authored by the loop's own
identity or matching its own branch prefix (`loop/{slug}/{run-shortid}`).
Every auto-fix commit is itself a new `push` delivery that re-matches the
trigger and spawns an **independent** new Task (not a continuation) — on top
of whatever phase-2 PR-webhook fan-out also fires for the same commit. Only
backstop is the 100-runs/day cap, whose exhaustion behavior (queued? dropped?
erroring?) is undocumented.
**Fix:** exclude the loop's own authored pushes / branch prefix from
re-matching triggers; reconcile push-trigger vs. phase-2 PR-webhook overlap
so one commit can't double-spawn; document cap-exhaustion behavior.

### 3. `overlap_policy` check-then-create has no transactional guarantee

**Section:** Run ("Overlap: before creating a task, check for an active run...").
No `SELECT ... FOR UPDATE`, advisory lock, or partial unique index is
specified. A schedule fire and a webhook fire landing within the async lag
between "TaskRun created" and "status flipped to running" (not just
instruction-level race — worker pickup lag widens this a lot) can both pass
the check. `skip` is silently violated; `cancel_previous` is worse (neither
fire sees a "previous" to cancel at check-time; cancellation only happens
retroactively, possibly after a PR is already opened).
**Fix:** state the concrete mechanism — Postgres advisory lock keyed on
`loop_id`, or a DB-enforced partial unique index on active-status runs per
loop — and define "active" precisely.

### 4. Dedup key is not scoped to the matched trigger/loop

**Section:** Run ("Idempotency"), GitHub event triggers §2 & §4.
The doc's own design has one webhook delivery matching **multiple**
`LoopTrigger` rows (§2), but the stated dedup key is the delivery GUID alone.
If two loops match the same delivery, the second loop's task creation gets
misidentified as a duplicate of the first and silently dropped — a
correctness bug on day one for any team running two loops off one repo/event,
not an edge case. Also unaddressed: dedup story for API-triggered (`POST
.../trigger/`) and manual runs — no idempotency-key mechanism at all for
PSAK callers retrying on timeout.
**Fix:** composite dedup key (`loop_trigger_id` + GUID / workflow id) enforced
by a DB unique constraint; add client-supplied idempotency key support for
API/manual triggers or explicitly state there is none.

### 5. `LoopTrigger` claims team-scoping it doesn't structurally have

**Section:** Data model.
The schema block lists `LoopTrigger` with only a `loop` FK — no direct
`team`/`team_id` column. The doc asserts "Both team-scoped (fail-closed
manager per repo conventions)," but `TeamScopedRootMixin`'s fail-closed
manager filters on a field declared directly on the model, not traversed
through an FK. This reintroduces, in a brand-new model, the exact
under-scoping pattern (`team` only as a derived property via `task.team`)
that the doc is simultaneously deleting from `TaskAutomation`. Per this
repo's AGENTS.md, this is a hard requirement, not a nice-to-have.
**Fix:** add a direct `team` FK on `LoopTrigger` (denormalized off
`loop.team`), consistent with other `products/tasks` models (`Channel`,
`TaskThreadMessage`).

### 6. Retention sweep can delete an in-flight run's task

**Section:** Run ("Cleanup").
"Keep the latest 200 per loop" is a pure recency rule with no run-status
awareness. A busy or long-lived loop can push an active run outside the
200-newest window while it's still executing. No statement of whether
soft-delete cancels the underlying Temporal workflow, or just orphans it
while breaking the loop-detail UI's promise to list it.
**Fix:** exclude non-terminal `TaskRun` statuses from the deletion candidate
set regardless of recency rank.

## 🟠 High

### 7. `LoopTrigger` matching may leak repo content across teams sharing a GitHub App installation

**Section:** GitHub event triggers §2.
The doc's own wording says installation → **teams** (plural) — one App
installation can map to multiple PostHog teams. Matching is described as
"`(repository, event, filters)`" with no mention of also constraining by
`github_integration_id`, and no stated validation that a `LoopTrigger`'s
`repository` string is actually in the enumerated repo list of its own
`github_integration_id`. If literal, Team B can type Team A's repo name into
its own trigger and receive Team A's issue/PR/comment content in its runs
and connector output — a real tenant-isolation break, not defense-in-depth.
**Fix:** scope matching to `(github_integration_id, repository, event,
filters)`; validate `repository` against the enumerable repo list of the
specific integration at trigger-creation time.

### 8. PSAK `loop:trigger` scope replays across every loop in the project

**Section:** API trigger auth.
PSAK scopes are project-wide by design (no object-level binding). A key
leaked from one loop's integration authenticates `POST
.../loops/<any-loop-id>/trigger/` for any loop in the project, with
attacker-controlled body becoming run context. Combined with finding 9, this
turns "one narrow-looking scope" into "fire anything in this project with a
chosen payload."
**Fix:** at minimum, document the shared-fate blast radius explicitly;
consider binding a minted key to a specific loop id.

### 9. "Fenced, not-instructions" is the sole stated defense against prompt injection into a write-capable agent

**Section:** Trigger payload handling, Security and guardrails.
A real, but known-bypassable, mitigation, treated as sufficient for a run
that combines attacker-influenceable input (GitHub issue/PR bodies on any
externally-contributable repo) with PR-write access and optionally
full-scope Slack/email/PostHog MCP. This is exactly the shape that has
produced real exfiltration/backdoor incidents in other agent products.
**Fix:** add a second layer — reduced connector/tool set by default for
trigger-fired runs, or require explicit opt-in before an externally-triggered
run gets write-connector access.

### 10. Full-scope loops can create persistent backdoor loops via MCP

**Section:** Connectors (MCP), Natural language creation.
The doc itself notes a full-scope loop can call `loops-create`. Composed with
9 and 8: an attacker who gets one full-scope loop to execute injected
instructions can plant a new, persistent loop with its own trigger and
connectors, running indefinitely as the same `created_by` identity,
independent of the original trigger being revoked.
**Fix:** scope "full" MCP grant for loops to loop-CRUD only, not the general
write surface; block loop-creation via MCP from within a triggered run
regardless of scope (no legitimate product reason for a scheduled run to
spawn new loops).

### 11. `TaskAutomation` deletion skips the mandated two-phase retirement workflow, and "unused" is asserted, not verified

**Section:** Takeover of TaskAutomation.
The doc's own capability table lists TaskAutomation's scheduling as
"Production." "Delete outright, no migration, no shim" contradicts this
repo's required retirement pattern (`SeparateDatabaseAndState`, two-phase
deploy) and doesn't state how "unused" was established (row count? usage
check?). Temporal Schedule cleanup ("deleted on deploy") names no mechanism,
no sequencing relative to the workflow-type rename, and no idempotency story
if cleanup fails partway. Mobile clients on old builds will hit hard 404s
against a deleted API with no deprecation window.
**Fix:** confirm `TaskAutomation.objects.count()` (or equivalent) before
committing to deletion; follow the mandatory `/django-migrations` retirement
sequence; name a concrete schedule-cleanup script sequenced before workflow
rename; state a mobile deprecation window.

### 12. Cross-repo agent-server contract has no version negotiation, and the sandbox already floats on `@latest`

**Section:** Workspace (multi-repo), Frontend, Phasing (Phase 5).
"Instead of `--repositoryPath`" reads as a hard cutover, with no capability
handshake — contrast with the existing `--baseBranch` fallback pattern
(blind retry-without-flag) that doesn't generalize to a structural protocol
change. Separately, `Dockerfile.sandbox-base` installs `@posthog/agent@latest`
with a cache-busted rebuild, decoupled from any posthog-repo deploy or
feature flag. A `PostHog/code` release that drops `--repositoryPath` support
before this repo's Phase 5 ships can break **every existing single-repo
run**, not just multi-repo loops, on the next unrelated image rebuild. "Existing
single-repo behavior unchanged" is only true at the Django/API layer, not
end-to-end.
**Fix:** pin `@posthog/agent` to an explicit version for the Phase 5 rollout
window; require agent-server to accept both invocation contracts for at
least one release; gate multi-repo loop exposure behind a feature flag tied
to a confirmed-live agent-server minimum version.

### 13. Multi-repo PR opening has no stated partial-failure policy

**Section:** Workspace (multi-repo).
No transactional/rollback semantics for the commit→push→PR-open sequence
across N repos. If repo A's PR opens and repo B's push fails (or a duplicate
branch collision on retry), repo A's PR looks fully green and mergeable in
isolation, understating that a coordinated cross-repo change is incomplete —
arguably worse than a fully-failed run because it looks like success in
repo A's context.
**Fix:** pick and state a policy — best-effort with an explicit
`needs_attention` surfacing per-repo status, or compensating rollback — and
make branch/PR creation idempotent against retries.

### 14. Webhook registry refactor touches shared, already-shipped production code with no isolation/rollout plan

**Section:** GitHub event triggers §1.
The registry generalization changes failure semantics for code Loops doesn't
own (conversations, tasks PR backstop, installation lifecycle) — no stated
per-handler exception isolation, no behavior-preserving-refactor-first
sequencing, no mention of the existing dual-route (`webhooks/github/pr` and
`webhooks/github`) mapping to the same view.
**Fix:** ship the registry refactor as a verified no-op change first (canary
against existing consumers), add the Loops handler as a separate, additive PR.

### 15. GitHub App event-subscription is a manual, unaudited ops step not gated in phasing

**Section:** GitHub event triggers §3, Phasing (Phase 3).
Listed as one bullet among four with no blocking relationship to code merge.
If it slips, `type=github` triggers for unsubscribed events (e.g. `push`)
appear fully configurable in the UI but silently never fire — hard to
root-cause. No mention of whether already-installed customer orgs need to
re-consent for new event types.
**Fix:** make the App subscription update an explicit blocking precondition
for Phase 3 enablement; enforce the v1 event allowlist server-side, not just
in the frontend.

### 16. Delivery-GUID dedup has no storage/TTL/scope design

**Section:** GitHub event triggers §4.
No existing pattern to model this on (confirmed: no dedup exists in
`github_webhook` or `products/tasks/backend/webhooks.py` today). Undefined:
storage backend, TTL, whether dedup is shared once at the dispatcher level or
reinvented per-handler, and fail-open vs. fail-closed behavior if the store
is unavailable.
**Fix:** Redis-backed, TTL bounded to GitHub's redelivery window, computed
once at the dispatcher level, explicit fail-open decision stated.

### 17. Trigger fan-out matching has no stated index/query plan, and `overlap_policy` doesn't protect against burst-of-distinct-events amplification

**Section:** GitHub event triggers §2.
`repository`/`event`/`filters` matching against JSON-config trigger rows,
scoped only after "resolve installation id to teams" — no promoted indexed
columns stated. Separately: a mass PR-sync burst (many distinct legitimate
deliveries on one repo) isn't bounded by `overlap_policy` at all, since that
policy only governs the _same loop_ being asked to fire while its own prior
run is still active — it does nothing for N different loops (or the same
loop matching N different PRs) firing concurrently in a tight window. The
100-runs/day cap bounds volume, not concurrency or rate.
**Fix:** state the query plan (team/integration-scoped lookup first, indexed
`repository`+`event` columns, `filters` evaluated last); add an explicit
per-loop concurrency/rate guard independent of `overlap_policy`.

### 18. No idempotent "PR already open" check for scheduled loops

**Section:** Workspace (multi-repo), Behaviors.
Every firing uses a distinct branch shortid; nothing checks GitHub for an
existing open PR from a prior firing before opening another. A daily loop
whose PR sits unmerged for a week produces 7 competing PRs against the same
files. `overlap_policy` only guards concurrent runs, not PR state persisting
across non-overlapping fires days apart.
**Fix:** before opening a new PR, check for and reuse/update an existing
open PR from the same loop against the same repo.

### 19. 100 runs/day cap is per-loop only; nothing bounds aggregate spend across a team's loops

**Section:** Security and guardrails.
No cap on number of loops per team, no aggregate runs/day, no stated
team/org scope for the cloud usage limit gate. A team facing the per-loop
cap has zero friction creating loop #2. The gate is the only real backstop
and the doc never states its scope.
**Fix:** state explicitly whether the usage gate is the real (team/org-level)
ceiling, or add an aggregate per-team cap.

### 20. Usage-limit gate rejections have undefined failure/notification semantics for unattended triggers

**Section:** Security and guardrails.
For `TaskViewSet.run`, gate rejection is synchronous UX a human sees
immediately. For schedule/webhook fires, nobody is watching — the doc
doesn't say whether a gate rejection counts toward `consecutive_failures`
(if not, auto-pause never engages for a loop silently blocked for days) or
whether it maps to any notification event kind (only in-app is
unconditionally on; push/email/Slack are opt-in).
**Fix:** wire gate rejections into `consecutive_failures`; make it a
`needs_attention`-class event, default-on for autonomous triggers.

## 🟡 Medium

- **Cooldown window value/scope/configurability entirely unspecified**, and
  likely conflicts with a per-channel `run_completed` subscription (a
  blanket per-(loop, kind) cooldown would silently drop most "wanted"
  per-run notifications unless scoped to failure/attention kinds only).
  _Notifications._
- **`needs_attention` event kind is referenced but never defined anywhere**
  — no stated condition produces it. _Loop / Notifications._
- **Retention default (200) and GitHub App v1 event list are each stated as
  decided prose in the body, then re-asked as unresolved in Open
  questions** — self-contradictory, pick one location. _Run + Open questions;
  GitHub triggers + Open questions._
- **`Task.loop` FK pattern unspecified** (`db_constraint=False`/NOT VALID vs.
  plain FK, `on_delete` behavior, indexing) despite `posthog_task` being a
  hot, high-write table. _Data model._
- **`Task.repositories` nullability/default/backfill unstated.** _Data model._
- **Auto-pause only catches failures, not expensive-but-technically-successful
  loops** (e.g. a loop that legitimately burns max iterations every run and
  reports success each time). _Security and guardrails._
- **`max_fix_iterations` has no server-side ceiling** — unlike the existing
  `MAX_CI_REPETITIONS` constant it's meant to generalize. _Loop primitives._
- **Multi-repo workspace cost not bounded per trigger scope** — a single-repo
  push still processes the full N-repo workspace, no cap on `repositories`
  list length. _Workspace (multi-repo)._
- **No stated behavior for cancellation mid-multi-repo-push under
  `cancel_previous`** — could produce the same partial-PR state as finding 13,
  triggered by policy instead of error. _Behaviors / Workspace._
- **Creator-deactivation guardrail reads as next-fire-only** — unclear
  whether an already-running sandbox/its minted credentials are killed
  immediately on deactivation, which matters because deactivation is often
  itself the security response. _Security and guardrails._
- **Mobile client version skew** on TaskAutomation API removal — no
  deprecation window or graceful-degradation story for un-updated builds.
  _Takeover of TaskAutomation / Frontend._
- **Phase 1 ships zero failure-visibility** for unattended cron loops —
  auto-pause and notifications are Phase 2, meaning a Phase-1 cron loop that
  fails every run is silent until someone manually checks. _Phasing._
- **Phase 1's net-new user value over already-Production TaskAutomation is
  thin** (model pinning + rename) — worth being explicit this phase isn't
  externally pitchable as "Loops v1." _Phasing._
- **`repositories` is multi-valued in the schema from Phase 1, but multi-repo
  execution doesn't land until Phase 5** — undefined validation behavior for
  a 2+-repo list submitted before Phase 5 ships. _Loop / Workspace._
- **Multi-repo CI-watch/fix scoping**: unclear whether a fix session that
  needs to touch repo B also re-touches an already-green repo A in the same
  shared sandbox session. _Workspace (multi-repo)._
- **Branch-protection drift not distinguished from a real CI failure** —
  `watch_ci`/`fix_review_comments` would keep attempting fixes against a
  policy gate (missing required review) the same as a code bug. _Behaviors._
- **Phase 6 run-chaining is in tension with the "not a DAG" non-goal** —
  conditional, event-triggered spawning of follow-up runs off prior runs'
  output is a small implicit state machine; not a violation of the letter
  of the non-goal, but the doc doesn't acknowledge the drift. _Behaviors /
  Non-goals._
- **JSON config fields have no schema-version story** for evolving
  `behaviors`/`connectors`/`notifications`/trigger `config` shape on
  already-stored rows. _Data model._

## 🟢 Low

- Possible check-then-act race on the 100-runs/day counter under webhook
  bursts if not atomically incremented.
- "Already supports N repos" (clone/storage layer) is used as evidence that
  understates how much of the multi-repo work is actually the
  agent-server invocation-contract layer, which is still single-repo today.
- No compatibility/soak/canary process specified for the one genuinely
  cross-repo wire contract (Phase 5), despite the rest of the doc applying
  that rigor everywhere else (auto-pause, usage gate, rate limits, cooldown).
- Open questions section doesn't state which questions block which phase.
- In-app notification "always" wording — unclear if it also bypasses the
  cooldown or just the per-channel `enabled` flag.

## Convergent findings (flagged independently by 2+ lenses)

- **Phase-2 babysitting's iteration bound is unenforceable as specified** —
  surfaced independently by the adversarial red-team lens (as the single
  biggest failure mode) and the cost/abuse lens (as the mechanical
  self-feeding push-trigger case). Two different reasoning paths, same root
  cause: no persisted counter across workflow boundaries.
- **`overlap_policy` is not a real concurrency guarantee** — surfaced by both
  the concurrency lens (TOCTOU on check-then-create) and the GitHub-infra
  lens (doesn't protect against burst-of-distinct-events amplification).
- **TaskAutomation deletion is under-specified** — surfaced by the data-model
  lens (skips mandated retirement workflow) and independently referenced by
  the security lens (mobile client skew) and product lens (phasing
  implications).
- **Cross-repo/agent-server coordination is the least-guarded change in the
  doc relative to its blast radius** — the only finding rated Critical by a
  lens whose entire mandate was sequencing risk, and it composes with the
  webhook-infra lens's "shared production code, no isolation plan" finding
  as the same underlying pattern: this plan is more careful about Loops'
  own new code than about the existing production surfaces it modifies or
  depends on.

## Recommended next step

Resolve the 6 Critical items above before Phase 1 engineering starts (items
1-2 are design gaps in the spec itself; items 3-6 are concrete
correctness/isolation bugs). The High-severity security findings (7-10)
should be resolved before Phase 4 (MCP surface) enables `posthog_mcp_scopes:
full` or GitHub triggers reach general availability. Everything else can be
tracked as implementation-time decisions, but the self-contradictions in
Open Questions (retention default, GitHub App event list) should be cleaned
up in the doc regardless, since they currently read as unresolved when the
body has already asserted an answer.
