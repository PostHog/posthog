# Engineering analytics

PR and CI lifecycle analytics for engineering teams, served as MCP tools and a read-only UI.

The differentiator: PostHog already tracks code at the bookends — generated by PostHog Code, observed in production by PostHog product analytics — but the middle (CI, review, merge, deploy) is invisible to both ends.
Engineering analytics is the middle, exposed through one query language (HogQL) and one primary surface (MCP) so human-driven agents and autonomous agents (e.g. PostHog Code) can call the same tools.

## Concepts

**PR** — an open GitHub pull request, draft or ready-for-review. The atomic unit of value.

**Draft phase** — experimentation. Low-friction expectations.

**Ready-for-review phase** — high stakes. Where this product measures aggressively (CI duration, review latency, cycle time).

**Workflow** — a GitHub Actions workflow run on a PR's head commit. One row in `github_workflow_runs`. v1 CI granularity; job-level breakdown is a later add.

**Bot** — an Author with `is_bot = True` (`handle.endswith("[bot]") OR handle in KNOWN_BOT_HANDLES`). Excluded from default cohorts; first-class in bot-impact analysis.

**Metric quality** — every tool return shape carries a `metric_quality: "precise" | "coarse" | "partial"` marker so callers can act on the result without paraphrasing what the metric measures.

## Data path

HogQL on `github_pull_requests` and `github_workflow_runs`.
No product Postgres DB.
No event ingestion in v1 — webhook ingestion in the GitHub source is the planned extension when state-transition timing matters.
