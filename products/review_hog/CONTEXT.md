# ReviewHog — glossary

Vocabulary settled while working on the product. Pure vocabulary — no spec, no implementation detail beyond what a term needs to be unambiguous.

- **Finding body** — the reviewer's description of a problem (`Issue.issue`, persisted as `ReviewIssueFinding.body`). Dual-audience text: it is rendered verbatim in the published comment AND consumed by three downstream LLM stages (the validator's `ISSUE` payload, dedup's fresh/prior finding payloads, and future turns' covered-findings block).
- **Validator argumentation** — the validator's why-valid / why-dismissed reasoning (`IssueValidation.argumentation`). For a _valid_ finding it is presentation-only (humans + the copy-paste fix prompt); its only pipeline consumer is dedup's `prior_ruling`, and only when the finding was _dismissed_.
- **Published copy** — what humans read on GitHub (inline comment sections, report body) and in the Code review UI. Today it is byte-identical to the pipeline payload; nothing requires that.
- **Pipeline payload** — the same finding/verdict fields as consumed by downstream LLM stages (validator, dedup, covered-findings). The quality-critical audience.
- **Structural conciseness vs compression** — _structural_: the same information content reshaped into labeled bullets (claim / trigger / evidence / impact); _compression_: dropping information to save words. The no-quality-loss constraint rules out compression of pipeline payloads; it does not rule out restructuring.
- **Verify, don't restate** — the rule for validator argumentation: it is the _verification delta_ (what was checked, what was found with file:line evidence, confirmed impact, priority rationale), never a restatement of the finding body's claim. Safe because the full body always travels with the argumentation — every consumer, human or LLM, receives both in the same comment or payload.

## Resolution stage (the "implement the fixes" flow — in design)

- **Resolution stage** — the ReviewHog-owned stage that runs after a review turn and drives the PR toward ready-to-merge: it loads the PR's posted comments and settles each one (implement / decline / escalate). It is part of the ReviewHog pipeline but independently startable on any in-scope PR — many PRs never get a review turn (too simple, StampHog-approved), so standalone mode is first-class, not a special case.
- **Comments-as-input invariant** — the stage's work-list is always the comments _posted on the PR_, never ReviewHog's in-memory findings. When chained after a review, ReviewHog's findings reach the stage the same way a human's do: as published comments. This keeps chained and standalone runs identical in shape.
- **In-scope PR** — the target set the resolution stage acts on: any **non-fork** PR in a repo the *acting team's* GitHub App installation can access, regardless of whether the PR's author is a person or a bot. Deliberately installation-relative, never org-specific: the stage is designed to work on any customer's project, so nothing in scope rules, prompts, or criteria may assume PostHog's own repos or internal docs — the session discovers the target repo's own conventions (CLAUDE.md / AGENTS.md / CONTRIBUTING, docs near the touched files) when present. Forks are excluded because we can't push commits to a branch we don't own and a fork head ref is attacker-influenced — the same fork guard ReviewHog already applies at fetch, now load-bearing because the resolution stage _writes_.
- **Comment-loading policy** — every comment on an in-scope PR is loaded regardless of who wrote it (human, ReviewHog, or any other bot). Author type never excludes a comment from consideration; it only feeds prioritization (see [[comment-source-rank]], to be settled under triage).
- **Work-list** — the unresolved review threads on the target PR, and nothing else. The unit of iteration is the **thread** (the whole conversation at one location), not the individual comment — the thing needing resolution is the thread's latest state. Outdated unresolved threads stay on the work-list (position drift usually means the code moved; "already addressed → reply + resolve" is a cheap, valuable outcome). Already-resolved threads are skipped. PR conversation comments and review bodies are **context, not work items**: the stage acts only on surfaces that can be both answered and resolved.
- **Resolution session** — the single warm, writable sandbox session per PR run: the PR head branch is checked out once, and the work-list is fed through it **one thread per turn**, in priority order. One session (not a per-thread fan-out) because the shared working tree is load-bearing — later turns see earlier turns' fixes, there is exactly one pusher (no push races), and priority order decides conflicts (the human's ask lands first).
- **Resolution turn** — one work-list thread processed in one session turn: investigate, judge worth/safe, implement + signed-commit when worth+safe, and return a structured verdict. Assessment and fix share a turn because they share the same repo investigation.
- **Verdict-driven side effects** — the sandbox agent never talks to GitHub's comment API; the driver (Temporal activity) posts replies and resolves threads server-side from the turn's structured verdict, mirroring ReviewHog's DB-driven publish. Code writes happen in-sandbox (signed commit to the PR branch); comment/resolve writes happen server-side from verdicts.
- **Deterministic pre-filter** — work-list trimming that needs no LLM call (e.g. a thread whose last reply is the stage's own with no activity since). Deterministic skips only; judgment stays in the session.
- **Thread outcome** — every work-list thread ends in exactly one of: **FIXED** (implemented + committed), **WON'T FIX** (deliberate decline with the reason), **ALREADY FIXED / OBSOLETE** (superseded — names the superseding change), or **ESCALATE** (worth doing but not safe unattended: reply, leave unresolved for a human). Taxonomy inherited from the interactive `triaging-pr-review-comments` skill; ESCALATE is the autonomous replacement for its ask-the-user gate.
- **Safe to fix now vs needs e2e** — the operational safety bar (inherited from the triage skill): a fix is safe when it is contained and provable in-session by unit tests/lint; asks touching LLM prompts, live external calls, or publish/workflow semantics need e2e verification → ESCALATE. Proportionality is part of the bar: a fix requiring new infrastructure (schema change, new abstraction, config knob) is a decision, not a mechanical fix → never auto-implemented.
- **Pre-verdicted thread** — a thread carrying a standing human verdict as a reply (e.g. **SAFE TO FIX** / **E2E REQUIRED**). The resolution stage honors these: SAFE TO FIX substitutes for the worth judgment when verification confirms it; E2E REQUIRED forces ESCALATE. This is the human override channel into an otherwise autonomous run.
- **Inventory** — the run's opening step: all unresolved threads fetched once and skimmed together before any turn, marking shared root causes. Turns stay strictly one-thread-each; a thread whose root cause an earlier turn already fixed closes as ALREADY FIXED pointing at that commit.
- **Resolution etiquette** — triage and fixing logic is identical for every thread regardless of author; only the *resolve* action is author-gated. Human-authored threads: reply as the bot with the outcome (fix + commit link, or the decline/escalate reasoning), **never resolve** — the human keeps the final word on their own thread. Bot-authored threads: reply and resolve on terminal outcomes. ESCALATE never resolves, for any author. (Autonomous adaptation of the triage skill's "bot threads: reply + resolve; human threads: hand back" invariant — the bot now speaks as itself, but resolution of human threads still belongs to humans.)
- **Entry points** — the durable interface is a trigger-style endpoint (mirroring `ReviewHogTriggerViewSet`: shared-secret bearer, server-side team/user resolution, unscoped mount), callable by API clients (other products, e.g. self-driving) and by a thin label-driven GitHub Action. Chained mode — `ReviewPRWorkflow` dispatching the stage after publish, per-run gated — is an internal dispatch, not an HTTP hop. A `run_resolution` management command mirrors `run_review` for dev/test. Deterministic skip gates run before any sandbox: closed/merged PR, fork (hard refuse — this stage writes), empty work-list after the pre-filter. One run per PR at a time via a deterministic workflow ID.
- **Writing mechanics** — one signed commit per FIXED thread (a signed commit is inherently remote — the harness creates commits via the GitHub API, so commit cadence _is_ push cadence), with that thread's reply/resolve posted right after its commit is visible. On a mid-run race (author pushed; signed commit refuses because the remote advanced): sync onto the new head once, re-verify, retry; if the branch moves again, yield gracefully — done threads keep their outcomes, the rest wait for the next run.
- **CI boundary** — the resolution stage handles review threads only; it never waits on, chases, or fixes PR checks. CI babysitting already exists (the Tasks CI follow-up loop). Known seam to close on the Tasks side: `_should_run_ci_follow_up` also fires on new unresolved threads and `DEFAULT_CI_MESSAGE` mandates comment-addressing — narrow that to checks-only (or skip its thread trigger) where the resolution stage is active, so two bots never fix the same comment.
- **Run budget** — a run resolves at most **20** threads, priority-ordered; the binding constraint is the warm session's context window (every turn accumulates, and fix quality degrades as it fills), not just sandbox cost. Overflow is always named in the run summary ("handled 20, N remain") and the next run continues — never silent truncation. Session/turn timeouts are inherited from the Tasks runner. Fix *size* is deliberately not a numeric cap — proportionality in the criteria skill is the guard. Model/effort pinned by `RESOLUTION_*` constants (validator's Claude default to start).
- **Per-thread watermark** — idempotency lives per thread, not per report: a thread whose latest `thread_verdict` still matches its current state (no comments newer than the verdict) is skipped deterministically; **any new reply re-opens triage for that thread**, so pushback on a WON'T FIX gets a fresh assessment with the pushback in context. Persistence home is the living `ReviewReport`: `thread_verdict` (new content schema, latest-wins per thread) + `commit` / `task_run` / `note` artefacts (their first writers).
