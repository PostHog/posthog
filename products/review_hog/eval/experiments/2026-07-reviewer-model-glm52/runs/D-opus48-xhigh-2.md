### [❌ dismissed] should_fix · security — products/review_hog/backend/receivers.py:144-156

**Webhook-leg opt-out gate resolve_stamphog_acting_reviewer is not fail-closed on error**  
_perspective: review-hog-perspective-contracts-security  ·  directly-related: True_

- **Problem:** resolve_stamphog_acting_reviewer is registered as the cross-product hook that stamphog's webhook path calls before re-reviewing a self-driving PR (leg 2). Its return value is the security gate: a user id authorizes stamphog to post a real GitHub approval on a bot-authored draft (a carve-out that relaxes the bot-author refusal); None must stop the review. Unlike handle_task_run_saved — which the module docstring/comments deliberately wrap in a blanket try/except so it 'must never raise' — this resolver calls _resolve_assigned_reviewer (which hits the DB via SignalReportArtefact query and resolve_org_github_login_to_users) and ReviewUserSettings.load (another DB read) with no exception handling. On a transient DB/resolution error the function raises instead of returning None. The design's stated invariant is that every scope check 'fails closed' (per the PR, 'missing resolver all fail closed'), but that was validated for an absent resolver, not a resolver that raises. If stamphog's webhook caller treats a raising hook as anything other than 'skip the review', a transient error becomes fail-open: stamphog approves a PR whose acting reviewer may have toggled the opt-in off — exactly the outcome the toggle exists to prevent. The gate should not depend on the caller's error handling to preserve the opt-out contract.
- **Suggestion:** Make the gate fail closed independently of the caller: wrap the body in try/except and return None on any exception (logging it), mirroring handle_task_run_saved's exception-proofing. For example:

    def resolve_stamphog_acting_reviewer(team_id, signal_report_id, task_created_by_id):
        try:
            acting_user_id = _resolve_assigned_reviewer(team_id, signal_report_id, task_created_by_id)
            if acting_user_id is None:
                return None
            if not ReviewUserSettings.load(team_id, acting_user_id).stamphog_review_inbox_prs:
                return None
            return acting_user_id
        except Exception:
            logger.exception("review_hog_stamphog_resolver_failed")
            return None

This guarantees that a transient failure suppresses the carve-out (no approval posted) rather than risking an approval on a PR whose reviewer opted out.
- **Validator:** - **Checked:** All runtime callers of the registered hook. Grepped the PR diff for `get_inbox_acting_reviewer_resolver` / `resolver(` — the resolver is invoked in exactly one production site, `_inbox_rereview_carve_out` (`products/stamphog/backend/tasks/tasks.py`, `acting_user_id = resolver(...)`), which is itself called from one place in `process_pull_request_event`. The initial leg `process_inbox_pr_review` does not call the resolver at all (the toggle was already checked in review_hog's receiver before queueing).
- **Found:** The sole caller already fails closed on a *raising* resolver, not just an absent one. In `process_pull_request_event`: `carve_out = _InboxCarveOut()` then `try: carve_out = _inbox_rereview_carve_out(...) except Exception as e: logger.exception(...); raise process_pull_request_event.retry(exc=e)`. - **Impact:** The finding's fail-open scenario cannot occur — the premise ('if the caller treats a raising hook as anything other than skip the review') is a conditional that the actual caller does not satisfy. Per the validation criteria this is 'already handled' (prevented at the parent caller, confirmed by reading it) and 'wrong/unreproducible'. Not worth surfacing.
