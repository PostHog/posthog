### [❌ dismissed] should_fix · security — products/stamphog/backend/tasks/tasks.py:113-121

**_parse_pr_url matches 'github.com/…/pull/N' as an unanchored substring anywhere in the string, not a validated GitHub PR URL**  
_perspective: review-hog-perspective-contracts-security  ·  directly-related: True_

- **Problem:** `_PR_URL_RE = re.compile(r"github\.com/([^/\s]+/[^/\s]+)/pull/(\d+)")` is used with `.search()`, which matches the pattern anywhere in the string rather than validating the URL's scheme and host. Because there's no anchor, a string like `"https://notgithub.com/owner/repo/pull/1"` or `"https://evil.example/redirect?to=github.com/owner/repo/pull/1"` parses successfully to `("owner/repo", 1)` even though the host isn't actually `github.com`. `_parse_pr_url` is the entry point for `process_inbox_pr_review`, which gates the self-driving inbox carve-out (bypasses bot-author refusal and the draft prerequisite downstream) — the same function class the PR's own security model treats as sensitive. The blast radius is currently limited because the resolved GitHub API call uses `repo_config.repository` from a team-scoped DB lookup rather than the regex-captured `owner/repo`, but the parsed `pr_number` from an unanchored match is trusted as-is, and this is the only validation standing between whatever string flows in as `pr_url` and 'fetch and self-driving-review PR #N in this team's configured repo'.
- **Suggestion:** Parse `pr_url` with `urllib.parse.urlparse` and validate `parsed.netloc == "github.com"` (or the configured enterprise host) plus an anchored path match (e.g. `re.fullmatch(r"/([^/]+/[^/]+)/pull/(\d+)/?", parsed.path)`) instead of an unanchored substring search, so a URL containing an embedded but not-actually-hosting `github.com/.../pull/N` segment can't be mistaken for a real GitHub PR link.
- **Validator:** - **Checked:** every caller of `_parse_pr_url`, the origin of the `pr_url` it parses, and every downstream use of its two return values (`repository`, `pr_number`).
- **Found (only caller):** `_parse_pr_url` is called in exactly one place — `process_inbox_pr_review` at `tasks.py:1130`. The webhook re-review carve-out the finding invokes as the sensitive surface (`_inbox_rereview_carve_out`, `tasks.py:144`) does NOT use it at all; it reads `pr.get("html_url")` from the verified webhook payload and gates on repo-native head + synced config + a team-scoped tasks-facade match + toggle re-check (`tasks.py:176-215`). So the bot-author/draft-bypass framing does not attach to this function.
- **Found (input is system-generated):** `pr_url` flows from `queue_inbox_pr_review` (`facade/api.py:128`) ← review_hog's `_start_stamphog_review` ← `output.get("pr_url")` in `receivers.py:93`, which the module docstring documents as "Written by the agent server when it observes the agent open the PR, or by the GitHub-webhook backstop." It is the real URL of a PR a PostHog Code run just opened, not free-form untrusted input crossing a boundary.
- **Found (host never used downstream):** in `process_inbox_pr_review`, `repository` is used solely as a lookup key against a team-scoped enabled config (`StamphogRepoConfig.objects.for_team(team_id)...filter(repository__iexact=repository, enabled=True...)`, `tasks.py:1141-1147`) — no match → silent no-op (`tasks.py:1151-1153`). The actual GitHub fetch uses `repo_config.repository` from that DB row, NOT the regex-captured owner/repo (`tasks.py:1159`). The URL's host/scheme is discarded after parsing.
- **Impact:** Anchoring the host closes no real hole. `pr_number` trust is identical whether the host is `github.com` or embedded in `notgithub.com` — a properly-anchored URL carries an equally-trusted number. In both cases the review target resolves to `repo_config.repository` PR #`pr_number`, gated by the team-scoped config lookup; a bogus/embedded host cannot redirect the fetch to another repo or team. The reviewer's own note concedes the fetch ignores the captured owner/repo, and the remaining `pr_number`-trust concern is unaffected by anchoring. Per the criteria this is speculative hardening against an input that can't occur as described and whose only real validation (team-scoped DB config + DB-repo fetch) is already present — precision-over-recall says drop.

### [❌ dismissed] should_fix · security — tools/pr-approval-agent/review_local.py:321-321

_perspective: review-hog-perspective-contracts-security  ·  directly-related: True_

- **Problem:** `self_driving=bool(context.get("self_driving_review"))` uses Python truthiness on a value pulled straight out of an externally-produced JSON blob. `self_driving` is not an ordinary display field — it is the single flag that relaxes the bot-author refusal and the draft prerequisite, i.e. it authorizes Stamphog to post a real GitHub approval on a bot-authored draft PR. `bool()` coercion means any JSON value other than `false`/`0`/`""`/`null`/absent is treated as `True` — critically, the JSON *string* `"false"` (as opposed to the JSON boolean `false`) evaluates to `True` under `bool()`, since any non-empty Python string is truthy. - **Suggestion:** Use a strict identity check for this specific flag given what it authorizes, e.g. `self_driving=context.get("self_driving_review") is True`, so any value other than the literal JSON boolean `true` fails closed rather than being coerced to enabled.
