### [✅ VALID] should_fix · performance — ee/hogai/tools/actions/tool.py:143-155

**DeleteActionTool re-fetches the action and re-checks object access twice on every approved delete**  
_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** `DeleteActionTool.format_dangerous_operation_preview` (lines 143-149) and `DeleteActionTool._arun_impl` (lines 151-155) each independently call `_fetch_action(self, action_id)` (a `database_sync_to_async(get_action_object)` DB round trip) and `await self.check_object_access(action, "editor", action="delete")`. Because `MaxTool._check_dangerous_operation` (ee/hogai/tool.py:339-349) is invoked from `_arun_with_context` before every call to `_arun_impl`, and LangGraph's `interrupt()` re-executes the entire tool node from scratch on resume (ee/hogai/tool.py:207-212, 351-404), a single user-approved deletion runs `format_dangerous_operation_preview` and then `_arun_impl` back-to-back in the same resumed invocation. That means every approved delete performs two separate `Action.objects.get(...)` queries for the same `action_id` plus two `UserAccessControl.check_access_level_for_object` calls, none of which are cached on `self` between the two methods even though they run in the same object instance and same execution. This is pure overhead on every delete: an avoidable extra DB round trip (each going through its own `database_sync_to_async` thread-pool hop) for a write path that should only need to fetch the row once.

This isn't hypothetical scale-only risk — it's guaranteed duplicate work on the happy path of every single delete, and the codebase already has a working precedent for exactly this: `UpsertDashboardTool` caches its per-invocation diff computation in `self._cached_update_diff` (ee/hogai/tools/upsert_dashboard/tool.py:604-605) specifically to avoid redoing work across `is_dangerous_operation` → `format_dangerous_operation_preview` → `_arun_impl`. `DeleteActionTool` doesn't apply the same idea to its own `_fetch_action` result.

- **Suggestion:** Cache the fetched action (and the fact that access was already checked) on the tool instance so the second call in the same execution is a no-op, e.g.:

```python
class DeleteActionTool(MaxTool):
    _cached_action: Action | None = None

    async def _get_checked_action(self, action_id: int) -> Action:
        if self._cached_action is not None and self._cached_action.pk == action_id:
            return self._cached_action
        action = await _fetch_action(self, action_id)
        await self.check_object_access(action, "editor", action="delete")
        self._cached_action = action
        return action

    async def format_dangerous_operation_preview(self, action_id: int) -> str:
        action = await self._get_checked_action(action_id)
        return (
            "Delete this action — it will be removed from any insights and funnels that use it:\n"
            f"{format_action_detail(action)}"
        )

    async def _arun_impl(self, action_id: int) -> tuple[str, None]:
        action = await self._get_checked_action(action_id)
        result = await database_sync_to_async(delete_action)(self._user, action)
        return result, None
```

This halves the DB round trips per approved delete and matches the caching pattern already established by `UpsertDashboardTool._cached_update_diff` for the same problem shape.

- **Validator:** Confirmed by tracing ee/hogai/tool.py: `_arun_with_context` (L207-212) calls `_check_dangerous_operation` (L339-349), which for a dangerous op calls `format_dangerous_operation_preview` before `_handle_dangerous_operation`'s `interrupt()` (L379). LangGraph re-executes the node function from scratch on resume, replaying deterministic code before the interrupt point; only the already-consumed `interrupt()` call short-circuits via the cached resume value. So on the resumed (post-approval) execution, `format_dangerous_operation_preview` runs again (fetch + access check) immediately before `_arun_impl` runs (fetch + access check again) — two `Action.objects.get(...)` calls plus two `UserAccessControl.check_access_level_for_object` calls in the same coroutine execution, on `self`, for the exact same `action_id`. This is a concrete, guaranteed-to-occur duplication on every approved delete, not a speculative edge case. Critically, the codebase already treats this exact call-chain shape (is_dangerous_operation → format_dangerous_operation_preview →_arun_impl, all potentially re-run together after a LangGraph resume) as worth optimizing: `UpsertDashboardTool` caches `self._cached_update_diff` (ee/hogai/tools/upsert_dashboard/tool.py:104, 604-605) specifically to avoid redoing work across those same three call sites. That precedent shows this isn't overengineering or paranoia — it's an established, low-risk convention in this codebase for this specific problem, and DeleteActionTool is the one new tool in this PR that has the dangerous-operation gate but doesn't apply it. The fix is a small, non-speculative change (cache the fetched+checked Action on the instance) with a real, quantifiable payoff (halves DB round trips on every approved delete).

_perspective: review-hog-blind-spots-general · directly-related: True_

- **Problem:** `CreateActionTool._arun_impl` and `UpdateActionTool._arun_impl` accept `name: str`/`name: Optional[str]` bound to `CreateActionToolArgs`/`UpdateActionToolArgs` (imported from `.core` and used as each class's `args_schema`), neither of which constrains `name`'s length. `Action.name` is `models.CharField(max_length=400, ...)` (products/actions/backend/models/action.py:43), which Postgres enforces as a hard `varchar(400)` column limit. The REST `ActionSerializer` never overrides `name` in its field list, so DRF's `ModelSerializer` auto-derives the field from the model and inherits the `max_length=400` validator automatically, returning a clean 400 validation error for an overlong name before it ever reaches the DB. This surfaces as a raw internal error rather than the actionable, retryable message the rest of this file is careful to produce for every other user-input mistake (blank name, duplicate name, not-found ID).
- **Suggestion:** Add a length constraint mirroring the model field, e.g. `name: str = Field(max_length=400, description=...)` on `CreateActionToolArgs`/`UpdateActionToolArgs.name` in core.py, so Pydantic validation catches the oversized input before it reaches `_check_name_available`/`action.save()` and produces the same clean validation-error path the executor already handles for schema mismatches, rather than an unhandled `DataError`.
- **Validator:** The technical premise checks out — `Action.name` is `CharField(max_length=400)` (products/actions/backend/models/action.py:43), `CreateActionToolArgs`/`UpdateActionToolArgs` in core.py don't constrain length, `_check_name_available` only checks blank/duplicate, and `Action.save()` (action.py:80-82) only calls `refresh_bytecode()` + `super().save()` with no `full_clean()`, so an overlong name would reach Postgres as an unvalidated write. But the issue's framing that this 'crashes with a raw DB error' overstates the actual impact: `_execute_tool_call` in ee/hogai/core/agent_modes/executables.py has a catch-all `except Exception as e` (L563-576) specifically designed as the 'Generic Exception: Unknown failures, treated as fatal (safety net)' documented in tool_errors.py's MaxToolError docstring. Any `DataError` from this path is caught there, captured to Sentry via `capture_exception`, and turned into a safe, non-crashing tool message telling the LLM not to immediately retry and to explain to the user — not a raw traceback or app crash. So this is already handled by an existing, intentional framework guarantee; the delta is UX polish (a specific 'name too long, shorten it' retry hint vs. a generic 'something went wrong' message), not a correctness or reliability failure. Separately, this gap is not something introduced by this PR's design — grepping sibling MaxTools shows the identical pattern everywhere: `UpsertDashboardTool`'s `name` field and `CreateNotebookTool`'s `title` field are also unconstrained `str`/`str | None` Pydantic fields with no `max_length` mirroring their models' CharField limits, and neither has ever been flagged or fixed. There's no codebase precedent establishing this as a convention the team enforces for MaxTool schemas (unlike the caching precedent found for the sibling double-fetch issue in this same file), so treating it as a PR-specific defect to fix here — rather than a pre-existing, systemic, low-priority characteristic of the whole MaxTool argument-schema pattern — doesn't hold up. Given the existing safety net plus the low likelihood of an LLM producing a 400+ character action name in practice, this falls closer to defensive-coding polish than a real, user-affecting bug.

_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** `ActionStepInput` lets the LLM set `selector`, `tag_name`, `text`, or `href` on a step with any `event` value (or none). - **Validator:** Verified against posthog/hogql/property.py:1220-1336 (`steps_to_expr`).
