### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:148-150,56-60

**Negative offset/limit crashes list_actions with an unhandled ValueError**  
_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** `list_actions` feeds `offset`/`limit` straight into a queryset slice with no lower-bound guard: `start = offset or 0` and `capped_limit = min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT)`, then `qs[start : start + capped_limit]`. Django rejects negative slice bounds — `QuerySet.__getitem__` raises `ValueError("Negative indexing is not supported.")` (django/db/models/query.py:410-417) when either the slice start or stop is negative. So a negative `offset` (e.g. -3 → `qs[-3:22]`) or a negative `limit` (e.g. -5 with offset 0 → `qs[0:-5]`) raises. The Pydantic args (`ListActionsToolArgs.limit`/`offset`, lines 56-60) have no `ge`/`gt` constraints, so nothing rejects negatives upstream, and `ListActionsTool._arun_impl` (tool.py:69-73) does not wrap `list_actions` in the `ActionToolError → MaxToolRetryableError` handling that the other tools use. The result is an unhandled exception surfaced as a hard, non-retryable tool failure rather than a graceful, correctable message to the agent. The equivalent REST list path explicitly guards this with `_parse_non_negative_int` (products/actions/backend/api/action.py:562-570), a safeguard this new path dropped.
- **Suggestion:** Constrain the inputs and/or clamp before slicing. Add bounds on the schema fields — `limit: Optional[int] = Field(default=None, ge=1, ...)` and `offset: Optional[int] = Field(default=None, ge=0, ...)` — and defensively clamp in `list_actions` to mirror the REST guard: `start = max(offset or 0, 0)` and `capped_limit = min(max(limit or DEFAULT_LIST_LIMIT, 1), MAX_LIST_LIMIT)`. That keeps the queryset slice in a valid range for any LLM-supplied value and turns malformed pagination into a no-op/normal response instead of a crash.
- **Validator:** The premise is correct and reproducible. `list_actions` (core.py:148-150) computes `start = offset or 0` and `capped_limit = min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT)` with no lower bound, then slices `qs[start : start + capped_limit]`. Django's `QuerySet.__getitem__` raises `ValueError("Negative indexing is not supported.")` for a negative start or stop, so `offset=-3` (→ `qs[-3:22]`) or `limit=-5` (→ `min(-5,100)=-5`, `qs[0:-5]`) crashes. `ListActionsToolArgs.limit`/`offset` (core.py:56-60) carry no `ge`/`gt` constraints, so Pydantic does not reject negatives upstream, and `ListActionsTool._arun_impl` (tool.py:69-73) — unlike create/update/delete — does not wrap the call in the `ActionToolError → MaxToolRetryableError` handling, so the exception surfaces as a hard, non-retryable tool failure. The REST list path explicitly guards this via `_parse_non_negative_int` (products/actions/backend/api/action.py:562-570), a safeguard this new path dropped. The trigger (LLM supplying a negative pagination value) and consequence (unhandled crash instead of a graceful/retryable message) are both concrete, and the fix is a trivial schema constraint + clamp. This fits the 'missing handling for a failure mode that will occur' bar. Likelihood of negative input is modest given the field descriptions steer toward 1-100/positive skip and the blast radius is a single recoverable failed tool call, but it is a real gap where an existing REST guard was not carried over, so should_fix is acceptable.

In any multi-environment project (a standard PostHog feature via Team.parent_team), self.\_team is a child environment team while actions live on the parent/root team, so list_actions returns empty/incorrect results and get/update/delete raise false MaxToolRetryableError('No action with ID X exists') for existing actions — including one create_action just saved. The 17 passing tests only mask this because they use single-team fixtures where team and team_id coincide. Concrete trigger and consequence, contradicts the established contract: a real correctness bug worth must_fix.
