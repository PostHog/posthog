This keeps the tool's validation contract aligned with the REST serializer and returns a model-fixable, retryable error instead of an unhandled `DataError`.

### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:148-150,56-60

_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** `ListActionsToolArgs.limit` and `offset` are declared as unconstrained `Optional[int]` (core.py:56-60), so the LLM (or a reconstructed approval payload) can pass negative values. `list_actions` then computes `start = offset or 0` (line 148) and `capped_limit = min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT)` (line 149) and slices `qs[start : start + capped_limit]` (line 150). A negative `offset` makes the slice start negative, and a negative `limit` (e.g. -5 with no offset) makes the slice stop negative. Django's `QuerySet.__getitem__` raises `ValueError("Negative indexing is not supported.")` for either case (confirmed in django/db/models/query.py). Because `ListActionsTool._arun_impl` (tool.py:69-73) calls `list_actions` with no try/except — and this `ValueError` is not an `ActionToolError` anyway — the exception propagates as a hard tool failure instead of a retryable, LLM-fixable message. The `min(..., MAX_LIST_LIMIT)` cap only guards the upper bound; the lower bound is unguarded. The sibling `ListFeatureFlagsToolArgs` already prevents this by constraining bounds at the schema level (`limit: int = Field(ge=1, le=100)`, `offset: int = Field(ge=0)`).
- **Suggestion:** Constrain the bounds at the schema level to match `ListFeatureFlagsToolArgs`: `limit: Optional[int] = Field(default=None, ge=1, le=MAX_LIST_LIMIT, ...)` and `offset: Optional[int] = Field(default=None, ge=0, ...)`. Alternatively (or additionally) clamp defensively in `list_actions`: `start = max(offset or 0, 0)` and `capped_limit = min(max(limit or DEFAULT_LIST_LIMIT, 1), MAX_LIST_LIMIT)` so malformed values degrade gracefully instead of raising.
- **Validator:** Verified. ListActionsToolArgs declares limit/offset as unconstrained Optional[int] (core.py:56-60), while the sibling ListFeatureFlagsToolArgs constrains them at the schema level (ge=1/le=100 for limit, ge=0 for offset — list_feature_flags.py:39-40), an established convention this diverges from. ListActionsTool.\_arun_impl (tool.py:69-73) wraps the call in no try/except, and ValueError is not an ActionToolError, so it surfaces as a hard tool failure rather than a retryable, LLM-fixable error. Trigger (LLM emitting a negative pagination value or miscomputing offset) is plausible; consequence is an unhandled exception; the fix (schema ge/le bounds, or clamping start/limit) is trivial and matches the sibling tool.
