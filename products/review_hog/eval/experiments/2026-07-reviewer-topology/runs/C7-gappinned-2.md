### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:34-45,105-121

**Element-match step fields (selector/text/href/tag_name) are silently ignored unless event is $autocapture**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** ActionStepInput exposes `selector`, `tag_name`, `text`, `text_matching`, `href`, `href_matching` as independent matching conditions, and their descriptions present them as standalone triggers ('CSS selector to match the target element', 'Element text content to match', etc.). But the HogQL compiler that turns steps into bytecode (`steps_to_expr` in posthog/hogql/property.py:1230) only emits selector/tag_name/href/text predicates when `step.event == "$autocapture"`. If the LLM builds a step with a selector (or text/href/tag) and no event — or any non-autocapture event — those conditions are dropped entirely. In the worst case, a step containing only a selector produces an empty predicate list and the compiler falls through to `ast.Constant(value=True)` (property.py:1330-1331), i.e. the action silently matches EVERY event. Compounding this, `_format_step` renders `selector=...`, `text ...`, etc. unconditionally, so `create_action`/`get_action` output echoes those fields back to the LLM as if they were active, hiding the discrepancy. The result is a tool that can confidently report it created a selector-scoped action while the compiled action matches all events — a silent correctness failure where behavior does not match intent.
- **Suggestion:** Make the autocapture requirement explicit so the LLM cannot construct a silently-broken step. Options: (1) document in the `selector`/`tag_name`/`text`/`href` field descriptions (and CREATE/UPDATE tool descriptions) that element-based matching only takes effect when `event` is `$autocapture`; and/or (2) validate in `to_step_dict`/`create_action`/`update_action` — if any of selector/tag_name/text/href is set while `event` is not `$autocapture`, raise an `ActionToolError` (retryable) guiding the LLM to set `event='$autocapture'`. Consider also having `_format_step` flag element conditions that will be ignored for a non-autocapture step so the tool output reflects the real compiled behavior.
- **Validator:** Verified against the codebase: `steps_to_expr` (posthog/hogql/property.py:1230) only emits selector/tag_name/href/text predicates inside the `if step.event == AUTOCAPTURE_EVENT` block (AUTOCAPTURE_EVENT = '$autocapture', constants.py:125), and a step whose predicates all get dropped falls through to `ast.Constant(value=True)` (line 1331), matching every event. There is no serializer or model validation coupling these element fields to autocapture. The new `ActionStepInput` (core.py:34-45) exposes selector/tag_name/text/href with descriptions that present each as a standalone trigger ('CSS selector to match the target element', etc.), with no mention that they only take effect for `$autocapture`. Because the LLM builds steps directly from these descriptions, it can plausibly emit a selector-only step with no event, producing an action that silently matches ALL events instead of the intended element scope.`\_format_step`(core.py:105-121) compounds this by echoing`selector=...` back unconditionally, so the tool reports a selector-scoped action while the compiled bytecode matches everything. Concrete trigger (LLM omits event=$autocapture on an element step) and concrete consequence (action silently matches all events) are both present, and the issue is directly introduced by this PR's tool surface. The lightweight documentation/validation fix is appropriately scoped. Keeping at should_fix: it doesn't break existing functionality but yields silently-wrong behavior on a realistic LLM path.

For example, have list_actions accept an optional access-control filter callback and wrap `Action.objects.filter(team=team, deleted=False)` with `user_access_control.filter_queryset_by_access_level(...)` exactly as posthog/api/routing.py:386 does for the list action. ### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:147-150

_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** `list_actions` pages with `qs.order_by("name")` and then slices `qs[start : start + capped_limit]`. `Action.name` is a nullable `CharField` with no unique constraint (confirmed in products/actions/backend/models/action.py:43 — no `unique=True`, and no unique index in migrations), so ties are common: multiple actions can share a name and unnamed actions all sort as NULL. Postgres does not guarantee a stable order among rows equal on the ORDER BY key, so successive paginated calls (which is exactly how this tool is meant to be driven — the description tells the LLM to use `limit`/`offset` to page) can skip or duplicate rows across page boundaries. For a discovery tool the agent relies on to enumerate actions before acting on them, this can cause it to miss an action entirely or process one twice.
- **Suggestion:** Add a unique tiebreaker to make ordering total and pagination deterministic, e.g. `qs.order_by("name", "id")`. `id` is the primary key and unique, so this guarantees a stable, repeatable order across paged calls with no meaningful cost.
- **Validator:** Verified: `Action.name` (products/actions/backend/models/action.py:43) is `CharField(null=True, blank=True)` with no unique constraint, and `list_actions` (core.py:147-150) does `qs.order_by("name")` then slices `qs[start:start+capped_limit]` for offset pagination. Postgres does not guarantee a stable row order among rows equal on the ORDER BY key, so ties — which arise here from NULL-named actions (all sorting as NULL) and any pre-existing duplicate names — can cause rows to be skipped or duplicated across page boundaries on successive paged calls. This is exactly how the tool is meant to be driven: the `limit`/`offset` descriptions instruct the LLM to page through results, and for a discovery tool the agent uses to enumerate actions before acting, skipping or double-processing an action is a genuine correctness concern.
