### [✅ VALID] must_fix · bug — ee/hogai/tools/actions/core.py:34-45

**Element-matching steps (selector/tag_name/text/href) are silently dropped unless event is "$autocapture", and element-only steps compile to match-all**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** `ActionStepInput` lets the LLM set `selector`, `tag_name`, `text`, and `href` on a step, and `create_action`/`update_action` store them verbatim (via `to_step_dict` → `Action.steps`). But the compiler that turns steps into the action's matching bytecode only honours those element matchers when the step's `event` is `$autocapture`. In `steps_to_expr` (posthog/hogql/property.py:1230) the selector/tag_name/href/text conditions live entirely inside `if step.event == AUTOCAPTURE_EVENT:` (`AUTOCAPTURE_EVENT = "$autocapture"`, posthog/constants.py:125). Consequences for actions created through these tools: (1) a step like `{selector: "button.cta"}` with no event produces ZERO match expressions, and `steps_to_expr` then appends `ast.Constant(value=True)` — the action matches EVERY event; (2) a step like `{event: "$pageview", selector: "..."}` silently ignores the selector and matches all `$pageview`s. Neither case sets `bytecode_error`, so nothing surfaces the problem, and `_format_step` compounds it by printing the dropped matchers (e.g. `selector='button.cta'`) as if they were active filters — so the agent reading the action back is told it is targeted when the compiled action is not. The tool descriptions actively advertise this footgun: `CREATE_ACTION_DESCRIPTION` (tool.py) says "Steps can match by ... CSS selector, element text/tag, href" and the field descriptions here never mention the `$autocapture` requirement. The frontend/REST path avoids this because element matching in the UI always sets `event` to `$autocapture`; the tool has no equivalent guard, and the tests only exercise `event="$pageview"`/`event="signed_up"` steps, so this path is untested.
- **Suggestion:** Enforce the autocapture invariant in the tool layer instead of silently producing a wrong action. Either (a) when any of `selector`/`tag_name`/`text`/`href` is set on a step, require `event == "$autocapture"` and raise `ActionToolError` otherwise, or (b) auto-set `event="$autocapture"` on such steps when no event is supplied (and reject a conflicting non-autocapture event). Also update the `selector`/`tag_name`/`text`/`href` field descriptions and `CREATE_ACTION_DESCRIPTION`/`UPDATE_ACTION_DESCRIPTION` to state that element matching only applies to `$autocapture` events, and consider having `_format_step` flag element matchers on a non-`$autocapture` step so the agent can see they are inert.
- **Validator:** Confirmed against the codebase. `steps_to_expr` (posthog/hogql/property.py:1230) only emits selector/tag_name/href/text conditions when `step.event == '$autocapture'`. A step with these matchers but a different event (or no event) drops them entirely, and the fallback at lines 1326-1331 appends `ast.Constant(value=True)` — so a selector-only step compiles to match-ALL-events, and a `{event:'$pageview', selector:...}` step matches all pageviews with the selector ignored. `Action.refresh_bytecode` (action.py:132-146) does not raise for this, so `bytecode_error` stays None and nothing surfaces the problem. `_format_step` (core.py:111-118) then renders the inert matchers as active filters, so the agent is told the action is targeted when it is not. The frontend/REST path avoids this because the edit UI (products/actions/frontend/components/ActionStep.tsx:82) only exposes element fields when the event is `$autocapture`; the tool's `ActionStepInput` has no equivalent guard and its field descriptions omit the requirement, while `CREATE_ACTION_DESCRIPTION` advertises element matching as a general capability. The trigger is concrete and reachable (the LLM is handed these fields and will set them), and the consequence is a silently wrong write — potentially an action matching every event. This is a genuine correctness defect in a write tool, not speculative. The suggested fix (auto-set/require `$autocapture` when element matchers are present, plus clarify descriptions) is appropriate.

### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:142-150

_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** `list_actions` replicates the REST list pagination but drops the non-negative guard the REST path applies. `ActionViewSet.list` sanitizes pagination with `_parse_non_negative_int` (products/actions/backend/api/action.py:562-605), which returns None for negatives; here `limit`/`offset` are plain `Optional[int]` on `ListActionsToolArgs` (no `ge=` constraint), so a (possibly prompt-injected) LLM can pass a negative value. A negative `offset` makes `start` negative, and a negative `limit` makes `capped_limit` negative via `min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT)`; either way the slice `qs[start : start + capped_limit]` hits Django's `QuerySet.__getitem__`, which raises `ValueError('Negative indexing is not supported.')` (verified at django/db/models/query.py:417). `ListActionsTool._arun_impl` (tool.py:69-73) does not wrap this call in the `ActionToolError`→`MaxToolRetryableError` mapping, so the agent gets an unhandled crash rather than a usable result. Separately, `limit or DEFAULT_LIST_LIMIT` silently turns an explicit `limit=0` into 25.
- **Suggestion:** Clamp the inputs before slicing, e.g. `start = max(offset or 0, 0)` and `capped_limit = min(max(limit or DEFAULT_LIST_LIMIT, 1), MAX_LIST_LIMIT)`, or constrain the pydantic schema fields (`offset: Field(default=None, ge=0)`, `limit: Field(default=None, ge=1, le=MAX_LIST_LIMIT)`) so out-of-range values are rejected as validation errors instead of crashing the tool.
- **Validator:** Verified. `list_actions` (core.py:148-150) computes `start = offset or 0` and `capped_limit = min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT)`, then slices `qs[start : start + capped_limit]`. A negative `offset` makes `start` negative; a negative `limit` makes `capped_limit` negative (the `min` with 100 preserves it), giving a negative slice stop — either raises Django's `ValueError('Negative indexing is not supported.')`. `ListActionsToolArgs` types both as plain `Optional[int]` with no `ge=` bound, so nothing rejects negatives, and unlike the create/update/get/delete tools, `ListActionsTool._arun_impl` (tool.py:69-73) does not wrap the call in the `ActionToolError`→`MaxToolRetryableError` mapping, so the agent gets an unhandled exception instead of a usable/retryable result. The REST equivalent explicitly sanitizes this with `_parse_non_negative_int` (products/actions/backend/api/action.py:562-577), showing the codebase treats out-of-range pagination from these same LLM/MCP consumers as a real concern. This is above the defensive-paranoia bar: the type permits the bad value, there is no upstream validation, the consequence is a concrete unhandled crash, and the fix (clamp or add `ge=` constraints) is trivial and matches existing convention. The secondary `limit=0`→25 observation is minor but the crash is a legitimate reliability defect.

### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/tool.py:69-73

**list_actions pagination sorts on a non-unique, nullable key — pages can skip or duplicate actions**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** ListActionsTool.\_arun_impl delegates paging to list_actions (core.py), which orders results with `qs.order_by("name")` and then slices `qs[start : start + capped_limit]` for each page. `name` is neither unique (there is no DB UniqueConstraint/unique_together on Action.name — uniqueness is only app-enforced in \_check_name_available, and only for non-blank names) nor non-null (`name = models.CharField(max_length=400, null=True, blank=True)`, and actions can be created without a name via the REST path or programmatically). Postgres does not guarantee a stable relative order for rows with equal sort keys across separate query executions, so when two or more actions share the same name — or, very commonly, when several actions have a NULL name and all collapse into one tie group — the physical order of those tied rows can differ between the query that returns page N and the query that returns page N+1. With LIMIT/OFFSET applied on top of that unstable order, a tied action straddling a page boundary can be returned twice or skipped entirely. Because this tool is explicitly the LLM's discovery mechanism ("Use this to discover an action's ID before ... updating, or deleting it"), a skipped action means the agent can conclude an action does not exist and fail to act on it, and a duplicated action wastes context and can cause double-processing. The single-column ordering is new code introduced by this PR; it does not inherit the REST viewset's (also weak, but full-list-by-default) ordering.
- **Suggestion:** Append a unique tiebreaker to make the ordering total and pagination deterministic, e.g. change `qs.order_by("name")` in list_actions (core.py:147) to `qs.order_by("name", "id")` (or `"name", "pk"`). The primary key is guaranteed unique and non-null, so equal/NULL names are then broken deterministically and each action appears on exactly one page regardless of how the underlying rows are physically stored.
- **Validator:** Verified: `Action.name` is `models.CharField(max_length=400, null=True, blank=True)` (products/actions/backend/models/action.py:43) with no `UniqueConstraint`/`unique_together` in `Meta` (only `indexes` and `db_table`), so name is both nullable and not DB-unique. `list_actions` (core.py:147-150) orders by the single column `name` and then applies `qs[start : start + capped_limit]` — LIMIT/OFFSET on a non-total ordering. Postgres does not guarantee a stable relative order for rows with equal sort keys (equal names, or the common case of several NULL-named actions collapsing into one tie group), and there is no index on `name` so results go through an unstable sort; when the underlying row order shifts between the page-N and page-N+1 queries — which is especially plausible here since the same agent also has create/update/delete tools that mutate the table mid-session — a tied action straddling a page boundary can be skipped or duplicated. Because this tool is explicitly the LLM's discovery path ("discover an action's ID before ... updating, or deleting it"), a skipped action can make the agent conclude the action does not exist. This is a real correctness defect (mishandled edge case that realistic inputs hit), the ordering is new code introduced by this PR, and the fix is the standard one-liner `order_by("name", "id")`. App-enforced name uniqueness reduces (but does not eliminate) same-name ties, and NULL names remain fully reachable, so should_fix is the appropriate severity.

_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** `create_action` (and `update_action`) enforce the advertised "unique within the project" guarantee purely with a read-then-write sequence: `_check_name_available` runs a `SELECT` (lines 184-189 / 195 / 218) and, only if it finds no clash, `action.save()` performs the `INSERT`/`UPDATE`. - **Validator:** The race is real but is a pre-existing, product-wide property of Actions that this PR does not introduce or worsen. So the issue's central 'contract divergence from the REST path' premise is mistaken: the tool faithfully mirrors the established, accepted REST behavior; uniqueness has always been best-effort here. The only new angle — an LLM emitting parallel `create_action` calls with an identical name in the same team inside the tiny check-then-save window — is speculative and low-probability, and the consequence is mild (a duplicate name the product already tolerates), not data loss or corruption. The robust fix (partial unique constraint + IntegrityError handling) is a schema migration affecting the whole product and the existing REST path, out of scope for a feature PR that replicates existing model behavior. A real-but-pre-existing, low-probability, low-impact limitation that isn't actionable against this change — the author would reasonably note the REST path is identical. Per precision-over-recall, drop.

This is not a high-QPS hot path — it is an agent discovery call invoked a handful of times per conversation, not per request at scale. The suggestion is explicitly conditional ('If action counts are expected to reach the documented thousands range...'), which is the definition of a speculative, future-proofing optimization: adding btree/pg_trgm indexes now for a scale most teams will never hit is premature optimization / overengineering. Per the precision-over-recall bar, a correct-but-negligible-impact performance note gated on an unlikely scale should be dropped rather than surfaced.
