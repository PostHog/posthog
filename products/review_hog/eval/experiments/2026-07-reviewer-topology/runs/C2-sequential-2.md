### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:34-45

**Element-matcher step fields (selector/tag_name/text/href) are silently ignored unless event is $autocapture, so Max can create ineffective or over-broad actions**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** In `steps_to_expr`/`action_to_expr` (posthog/hogql/property.py:1230), the element-matching conditions — `selector`, `tag_name`, `href`/`href_matching`, `text`/`text_matching` — are only compiled into the action's bytecode inside the `if step.event == AUTOCAPTURE_EVENT:` block. If a step supplies these fields but `event` is anything other than `$autocapture` (including `None` or a custom event name), those conditions are dropped entirely. A step with only a `selector` and no `event`/`url` compiles to an empty expression, which matches EVERY event. The tool surface actively invites this mistake: `ActionStepInput` (core.py:34-45) documents `selector`, `tag_name`, `text`, `href` as standalone matchers with no mention of the `$autocapture` requirement, and `CREATE_ACTION_DESCRIPTION` (tool.py:47-51) says steps "can match by event name, URL, CSS selector, element text/tag, href". Bytecode still compiles cleanly, so `_format_action` shows no `bytecode_error` and even echoes `selector=...`, giving the LLM (and user) false confidence that the created action does what was asked while it silently matches all events or nothing meaningful.
- **Suggestion:** Either (a) validate in `to_step_dict`/`create_action`/`update_action` that any step using `selector`/`tag_name`/`text`/`href` also sets `event="$autocapture"`, raising `ActionToolError` with a clear message otherwise, or (b) make the field descriptions and `CREATE_ACTION_DESCRIPTION` state explicitly that element-based matchers only take effect when the step's `event` is `$autocapture`. Option (a) is preferable since it converts a silent no-op into a retryable, self-correcting error for the agent.
- **Validator:** Confirmed against the code. In steps_to_expr (posthog/hogql/property.py:1227-1282), the element matchers selector/tag_name/href/text are compiled only inside `if step.event == AUTOCAPTURE_EVENT:`. So a step supplying a selector but with event=None (or a custom/non-autocapture event) drops those conditions entirely; a step with only a selector reduces to an empty exprs list and matches every event. This is a reachable, foreseeable failure for an LLM-driven surface: ActionStepInput (core.py:34-45) documents selector/tag_name/text/href as standalone matchers and CREATE_ACTION_DESCRIPTION (tool.py:50) says steps 'can match by ... CSS selector, element text/tag, href' with no mention of the $autocapture prerequisite, actively inviting the mistake. Worse, _format_step (core.py:111-118) echoes `selector=...` unconditionally and bytecode compiles cleanly (no bytecode_error), so both the LLM and user get false confidence that the action does what was asked while it silently matches all events. The concrete trigger and consequence are both nameable, the condition genuinely occurs (LLMs routinely omit domain prerequisites they aren't told about), and it is directly related to the tool surface this PR adds. Either fix is cheap: validate that element-based steps set event=$autocapture and raise ActionToolError (preferred, converts a silent no-op into a retryable self-correcting error), or state the $autocapture requirement in the field descriptions and CREATE_ACTION_DESCRIPTION.

### [✅ VALID] should_fix (validator→consider) · best_practice — ee/hogai/tools/actions/core.py:56-60,148-150

**list_actions lacks lower-bound validation on limit/offset; negative values raise an uncaught ValueError**  
_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** `ListActionsToolArgs.limit`/`offset` are typed `Optional[int]` with no range constraint, and `list_actions` feeds them directly into the Django slice `qs[start : start + capped_limit]` where `start = offset or 0` and `capped_limit = min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT)`. A negative `offset` makes `start` negative, and a negative `limit` makes the slice stop negative — both cause Django to raise `ValueError("Negative indexing is not supported.")` (confirmed: offset=-1 → start=-1, limit=-1 → stop=-1). LLMs plausibly pass values like `offset=-1`. Unlike `GetActionTool`/`CreateActionTool`/`UpdateActionTool`, `ListActionsTool._arun_impl` (tool.py:69-73) wraps the call in no try/except, so this surfaces as a hard, non-retryable tool error rather than a correctable message. This is also a contract inconsistency with the REST equivalent: `ActionViewSet.list` deliberately clamps these via `_parse_non_negative_int` (products/actions/backend/api/action.py), which ignores negative values instead of crashing.
- **Suggestion:** Validate the boundaries at the entry point, mirroring the REST path. Add Pydantic constraints so invalid input is rejected before `_arun_impl` runs — e.g. `limit: Optional[int] = Field(default=None, ge=1, le=MAX_LIST_LIMIT, ...)` and `offset: Optional[int] = Field(default=None, ge=0, ...)` — and/or clamp defensively inside `list_actions` (`start = max(0, offset or 0)`, `capped_limit = max(1, min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT))`) so a negative value can never reach the slice. Clamping matches the REST `_parse_non_negative_int` behavior and keeps the tool result graceful.
- **Validator:** The premise checks out. In list_actions (core.py:148-150) `start = offset or 0` and `capped_limit = min(limit or DEFAULT_LIST_LIMIT, MAX_LIST_LIMIT)` are fed straight into the Django slice `qs[start : start + capped_limit]`. offset=-1 yields start=-1 (negative index) and limit=-1 yields capped_limit=-1 (negative stop), both of which make Django raise ValueError('Negative indexing is not supported.'). ListActionsToolArgs (core.py:56-60) applies no ge/le constraints, and ListActionsTool.\_arun_impl (tool.py:69-73) — unlike its sibling tools that wrap ActionToolError into MaxToolRetryableError — has no try/except, so the ValueError propagates as a hard, non-retryable tool error. Negative pagination values are a plausible (if not frequent) LLM input, and the REST equivalent (ActionViewSet.list via \_parse_non_negative_int) deliberately clamps rather than crashing, so there is a genuine contract inconsistency. The trigger and consequence are both concrete and the fix is a one-line Pydantic constraint or a max()/clamp, matching an established pattern. This clears the noise bar: it is not a false premise, not overengineering, and not unreachable. That said, the real-world impact is limited — no data loss or corruption, just a degraded error message on unusual input that a well-behaved LLM would rarely produce — so it is a minor robustness/consistency gap rather than a should_fix defect.

### [✅ VALID] must_fix · bug — ee/hogai/tools/actions/core.py:143-143,171-171,198-198

**Action tools filter by team= object, bypassing RootTeamMixin project-scope rewriting — actions become invisible/unfetchable in secondary environments**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** `Action` is a `RootTeamMixin` (project-scoped) model: `Action.save()` reassigns the row to `self.team.parent_team` when a parent exists, and `RootTeamQuerySet.filter` only rewrites the lookup to the root/parent team when it is filtered by the `team_id=` keyword — it explicitly checks `if "team_id" in kwargs` and otherwise falls straight through to `super().filter(...)`. The new tool code is inconsistent: `list_actions` (line 143 `Action.objects.filter(team=team, ...)`), `get_action_object` (line 171 `Action.objects.get(team=team, ...)`), and the `create_action` count (line 198 `Action.objects.filter(team=team, ...)`) all filter by `team=<object>`, which does NOT trigger the parent-team rewrite, while `_check_name_available` (line 184) uses `team_id=` (which does) and `save()` stores the row under the root team. The REST `ActionViewSet` avoids this by using `team_id=self.team_id` everywhere. The conversation API passes the request's environment team (`team=self.team`) into the tools, which for any project with more than one environment is a child team with `parent_team` set. Result: from a secondary environment, `list_actions` returns "No actions found" even when the project has many actions (they are stored under the root team), `get_action`/`update_action`/`delete_action` raise not-found for every real action, and `create_action` "succeeds" but writes the row under the root team so the same toolset can never read it back. The behavior is completely broken in multi-environment projects, and the existing tests never exercise it because their teams have `parent_team=None`.
- **Suggestion:** Filter by `team_id=self._team.id` (or thread a `team_id: int` through) in `list_actions`, `get_action_object`, and the `create_action` count query, matching the REST `ActionViewSet` (`team_id=self.team_id`) and the `_check_name_available` name check. This routes reads through `RootTeamQuerySet`'s parent-team rewrite so they resolve to the same root team the rows are actually stored under. Add a test with a secondary environment (a team whose `parent_team` points at the project's root team) covering create-then-get and list, to lock in the fix.
- **Validator:** Verified end to end. Action inherits RootTeamMixin (products/actions/backend/models/action.py:42), whose save() reassigns the row to self.team.parent_team when a parent exists (posthog/models/utils.py:534-535), and whose RootTeamQuerySet.filter only applies the parent/root-team rewrite when the literal keyword `team_id` is present — `if "team_id" in kwargs:` (utils.py:484), otherwise it falls straight through to super().filter (utils.py:505). Filtering by `team=<object>` puts the key `team` (not `team_id`) in kwargs, so no rewrite happens and the query resolves to the child environment's id. The tool code is internally inconsistent about this: list_actions (core.py:143), get_action_object (core.py:171, get() delegates to filter() so it is affected too), and the create_action count (core.py:198) all use `team=team`, while_check_name_available (core.py:184) uses `team_id=` and save() stores under the root team. The REST ActionViewSet deliberately uses `team_id=` everywhere (safely_get_queryset line 553, create count line 240), confirming this is the required pattern for this project-scoped model. parent_team is a real, populated FK (team.py:283). Consequence in any multi-environment project accessed from a secondary (child) environment: list returns nothing, get/update/delete raise not-found for every real action, and create appears to succeed but writes the row under the root team where the same toolset can never read it back — a silent, complete break of the feature with confusing write-then-can't-read behavior. The trigger (secondary environment) and consequence are both concrete and reachable for real users of the environments feature, and the existing tests miss it because their teams have parent_team=None. This is a genuine correctness bug and a contract inconsistency with the REST path; the fix (thread team_id and filter by it) is trivial. must_fix is appropriate given it silently breaks the entire toolset for affected projects.

### [❌ dismissed] should_fix · performance — ee/hogai/tools/actions/core.py:143-150

**list_actions loads and deserializes the unused bytecode JSONField for every listed action**  
_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** The list queryset `Action.objects.filter(team=team, deleted=False)` (line 143) fetches full rows including the `bytecode` JSONField, which stores compiled HogQL (a flat list of opcodes/operands that can run to several KB per action). Django deserializes `JSONField` values via `from_db_value` for every fetched row, so up to `MAX_LIST_LIMIT` (100) bytecode blobs are parsed into Python and then immediately discarded — `_format_action(..., detailed=False)` only reads `id`, `name`, `description`, `steps_json` (via the `.steps` property) and `bytecode_error`, and never touches `.bytecode` (confirmed: the only `.bytecode` read in the module tree is in the test file). This is wasted DB transfer plus JSON-parse CPU on the hot path of a tool the PR explicitly designs for projects with 'thousands of actions' and whose description instructs the LLM to page repeatedly via `limit`/`offset`, so the cost is paid on every page.
- **Suggestion:** Defer the unused column on the list queryset only: `qs = Action.objects.defer("bytecode").filter(team=team, deleted=False)`. Keep `bytecode_error` loaded (it is rendered). Do NOT add `.defer("bytecode")` to `get_action_object` (line 171), which feeds the update/delete paths — `Action.save()` calls `refresh_bytecode()` which reads `self.bytecode` (`if new_bytecode != self.bytecode ...`), so a deferred field there would trigger a per-instance lazy reload query, reintroducing an extra round-trip. Since `list_actions` uses its own queryset and never saves the fetched rows, deferring there is safe.
- **Validator:** The technical premise is accurate — list_actions with detailed=False only reads id/name/description/steps/bytecode_error and never touches .bytecode, so the fetched-but-unused bytecode JSONField is deserialized and discarded, and deferring it on the list queryset (but not on get_action_object, which feeds save()/refresh_bytecode) would be safe. But it does not clear the performance bar of 'bites at real scale.' This is not an N+1, an unbounded loop, a missing index, or blocking I/O on an async path — it is a single query bounded at MAX_LIST_LIMIT=100 rows fetching one extra column. The tool is invoked by an LLM, where the model round-trip (seconds) dwarfs the cost of transferring and JSON-parsing at most 100 bounded blobs by orders of magnitude, so the relative impact is negligible. Decisively, the actual production hot path — ActionViewSet.list, which serves the real actions list page to every UI user — uses `Action.objects.select_related('created_by').all()` with no `.defer('bytecode')`. If the team does not bother deferring bytecode on the genuinely high-traffic UI list, a capped, low-frequency LLM tool certainly does not need it. Per precision-over-recall, this is a valid-but-immaterial micro-optimization that is not worth surfacing.
