**list_actions output size is unbounded despite the result-count cap**  
_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** The module comment (lines 13-14) and PR description claim list*actions is "bounded by construction" so it "can never blow up the agent's context window." That guarantee only holds for the \_number* of rows (capped at MAX*LIST_LIMIT=100), not the \_size* of the rendered output. In the non-detailed list view, `_format_action` inlines every step of every returned action via `'; '.join(_format_step(s) for s in steps)`, and `_format_step` embeds the full, user-controlled `url`, `selector`, `text`, and `href` strings. An action can have many steps, each with long selector/URL strings; multiplied by up to 100 actions per call, the returned string can be very large and unpredictable, defeating the stated context-protection goal. Because the LLM calls this repeatedly during discovery, a few actions with large step definitions can flood the agent's context window.
- **Suggestion:** Bound the per-action output in the non-detailed (list) view. For example, in `_format_action` when `detailed=False`, render only the step count and/or a short summary of the first step, or cap the joined step string to a fixed number of characters with an ellipsis, e.g. `lines.append(f"  steps: {len(steps)}")` plus a truncated preview. This keeps total list output bounded regardless of how complex individual actions are, while `get_action` (detailed=True) still exposes the full definition for a single action.
- **Validator:** The mechanism is real and correctly described: in the non-detailed list view (core.py:134), \_format_action joins the full rendering of every step, and \_format_step embeds the full user-controlled url/selector/text/href strings. So the rendered output size is bounded by rows × steps × string length, not by the MAX_LIST_LIMIT row cap alone — the module comment's 'bounded by construction' claim is technically overstated. However, the practical impact is limited: the default limit is 25 (max 100), real actions typically carry only 1–3 steps with moderate-length selectors/URLs, and the LLM can narrow results via `search`. Producing genuinely context-flooding output requires an atypical combination of many actions each with many long-selector steps; a 'few actions' will not flood the window as the finding implies. The premise is not wrong and the suggested compaction of the list view is a reasonable improvement, so this is a legitimate but minor design observation rather than a concrete bug — should_fix overstates it. Keeping it on record but downgrading to consider so it is not surfaced as a blocking finding.
