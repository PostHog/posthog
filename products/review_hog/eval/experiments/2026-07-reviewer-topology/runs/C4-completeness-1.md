**Action name length not validated; diverges from REST contract and raises uncaught DB error**  
_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** `_check_name_available` only rejects blank names; it does not bound the name length. `Action.name` is `CharField(max_length=400)` (a Postgres `varchar(400)` column). The REST contract enforces this limit at the serializer layer (the model-derived `name` field), returning a clean 400 for over-long names. The tool path constructs `Action(...)` and calls `.save()` directly, and Django's `.save()` does not run field validators, so an LLM-supplied name longer than 400 characters is passed straight to Postgres and raises `django.db.utils.DataError` (`value too long for type character varying(400)`). That exception is not an `ActionToolError`, so it is not mapped to a retryable error in `create_action`/`update_action` — it propagates as an unhandled failure rather than a graceful, model-fixable message. This is a missing input-boundary check that the REST contract guarantees.
- **Suggestion:** Enforce the same 400-character bound the model/serializer contract defines. Either add `max_length=400` to the `name` fields on `CreateActionToolArgs`/`UpdateActionToolArgs` (Pydantic will validate before execution and surface a clean schema error), or add a length check in `_check_name_available` (e.g. `if len(name) > 400: raise ActionToolError("Action name must be at most 400 characters.")`) so the failure becomes a retryable, self-correctable error consistent with the REST path.
- **Validator:** The technical premise is largely accurate: `_check_name_available` bounds only blank names, `Action.name` is a varchar(400), Django's `.save()` skips field validators, and `create_action`/`update_action` catch only `ActionToolError`. So a >400-char name would raise `django.db.utils.DataError`. However, the impact is minimal and the trigger is practically unreachable. Action names are short human labels ('Signup', 'CTA click'); an LLM producing a 400+ character action name essentially never happens in real use, placing this squarely in the 'never-gonna-happen edge case' bucket the validation criteria say to drop. The consequence is also mild: no data loss, no security or tenant-isolation impact, no corruption — the write simply fails. The premise's claim of an 'unhandled failure' is overstated: `tool_errors.py` documents that a generic Exception is caught as a fatal safety-net error, so the agent doesn't crash; it merely surfaces a fatal (non-retryable) message instead of a retryable one. That retryable-vs-fatal divergence for a name no LLM will realistically generate is not worth surfacing. Following precision-over-recall, drop it.

**Element matchers (selector/tag_name/text/href) silently compile to match-all unless event is $autocapture**  
_perspective: review-hog-completeness-gap · directly-related: True_

- **Problem:** The `ActionStepInput` fields `selector`, `tag_name`, `text`/`text_matching`, `href`/`href_matching` are only translated into bytecode when the step's `event == "$autocapture"`. So a step that sets, e.g., `selector="button.cta"` but leaves `event` unset (or a custom/non-autocapture event) has its element matchers dropped during compilation. Worse, if that dropped matcher was the step's only condition, `exprs` ends up empty and the compiler appends `ast.Constant(value=True)` (property.py:1330-1331) — the step then matches EVERY event. Nothing in the field descriptions or the `create_action`/`update_action` tool descriptions tells the model that element matchers require `event="$autocapture"`. An agent asked to "create an action for clicks on button.cta" will naturally set `selector` alone, and `create_action` echoes the selector back in its success output (`format_action_detail`), so both the agent and user believe a targeted action was created when the compiled action actually fires on all events. The frontend REST path avoids this because its UI forces `event=$autocapture` before exposing element fields; the raw-JSON tool surface has no such guardrail.
- **Suggestion:** Either (a) auto-default `event` to `$autocapture` when any element matcher (selector/tag_name/text/href) is provided and event is unset, inside `to_step_dict`/`create_action`/`update_action`; or (b) validate and raise `ActionToolError` when an element matcher is set without `event="$autocapture"`; and (c) at minimum, document the requirement explicitly in the `selector`/`tag_name`/`text`/`href` field descriptions and in `CREATE_ACTION_DESCRIPTION`/`UPDATE_ACTION_DESCRIPTION` so the model sets the event. Approach (a) matches user intent most closely.
- **Validator:** Verified against the live codebase and the claim holds. In `posthog/hogql/property.py` (steps_to_expr), the entire element-matching block (selector, tag_name, href, text) is gated behind `if step.event == AUTOCAPTURE_EVENT:` (line 1230). When `len(exprs) == 0` the compiler appends `ast.Constant(value=True)` (line 1331), so a step. The `Action` model's `save()` → `refresh_bytecode()` → `action_to_expr` → `steps_to_expr` performs no normalization that would auto-set `event=$autocapture` when element matchers are present, so nothing upstream neutralizes this. The trigger is concrete and plausible: an agent told to 'create an action for clicks on button.cta' will naturally set `selector` alone (the raw-JSON tool surface has no UI forcing event=$autocapture as the frontend does), and none of the field descriptions, CREATE_ACTION_DESCRIPTION, or UPDATE_ACTION_DESCRIPTION state that element matchers require `event="$autocapture"`. The consequence is a silent correctness bug: a step meant to match specific clicks compiles to match EVERY event, and`format_action_detail` echoes the selector back so both agent and user believe a targeted action was created. This meets the keep bar (nameable trigger + severe silent consequence) and is a real completeness gap worth surfacing. The reviewer's 'should_fix' priority is appropriate — it is a silent misbehavior but recoverable and not data-loss/security.

### [❌ dismissed] must_fix · security — ee/hogai/tools/actions/core.py:142-150

**list_actions bypasses object-level access control, exposing restricted actions**  
_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** `list_actions` fetches `Action.objects.filter(team=team, deleted=False)` and the `ListActionsTool` (tool.py:61-73) only performs a resource-level check via `get_required_resource_access() -> [("action", "viewer")]`. It never applies object-level access filtering. `action` is an object-level access-controlled resource (it is in `ACCESS_CONTROL_RESOURCES` and `model_to_resource(Action)` returns `"action"`), and the REST `ActionViewSet.list` path enforces this by running `self.user_access_control.filter_queryset_by_access_level(queryset)` (routing.py:362/386), which excludes actions the user is explicitly blocked from (per-object `AccessControl` with level `none`) and, when the user lacks resource access, restricts results to explicitly-allowed/owned objects. This PR added `check_object_access` to get/update/delete (tool.py:86/124/145/153) in response to earlier review, but list was left unguarded. Result: a user with default resource-level `viewer` but an explicit object-level `none`/restricted `AccessControl` on a specific action can still read that action's name, description, and full step (trigger) definitions — and even the total count — through the `list_actions` Max tool, which the REST list path hides. This is an object-level data-exposure gap and is inconsistent with the single-object tools in this same PR.
- **Suggestion:** Filter the list queryset by object-level access the way the REST path does. Thread the tool's `UserAccessControl` into `list_actions` and apply `user_access_control.filter_queryset_by_access_level(qs)` to the base queryset (before `total = qs.count()` so the count is also correct), or otherwise exclude blocked action IDs. For example, have `ListActionsTool._arun_impl` pass `self.user_access_control` (wrapped in `database_sync_to_async`) into `list_actions`, and inside `list_actions` do `qs = user_access_control.filter_queryset_by_access_level(qs)` right after building the base `team=team, deleted=False` queryset. This makes `list_actions` consistent with `get_action`/`ActionViewSet.list` and closes the read-side leak.
- **Validator:** The reviewer correctly established that `action` is in ACCESS_CONTROL_RESOURCES and that the REST list path applies `filter_queryset_by_access_level` (routing.py:362), while `list_actions` only does a resource-level check. However, the claimed leak scenario is not reachable for actions specifically. `minimum_access_level("action")` returns "viewer" (user_access_control.py:169-170) and `default_access_level("action")` returns "editor", and `AccessControlSerializer.validate_access_level` (ee/api/rbac/access_control.py:108-112) rejects any access control set below the resource minimum. So an object-level `none`/blocking AccessControl cannot be created for an action — every user with action resource access is guaranteed at least viewer on every action. In `filter_queryset_by_access_level`, `blocked_resource_ids` is only populated by object-level `none` controls, which are impossible here; thus for a user with resource-level access the REST list returns the full queryset unfiltered, identical to `list_actions`. Consequently there is no data-exposure gap: adding `filter_queryset_by_access_level` would be a no-op for actions, and the get/update/delete object checks are meaningful only for their `editor` level, not the always-satisfied `viewer` a list check would use. This is a deliberate design (actions are project-shared building blocks that cannot be individually hidden). The premise of the must-fix security finding is mistaken, so it should be dropped.

_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Suggestion:** Add a unique tiebreaker to make ordering deterministic: `qs = qs.order_by("name", "id")`. The trailing `id` guarantees a total order so offset/limit paging is stable and lossless even when names collide or are null.
- **Validator:** Verified: `list_actions` orders by `qs.order_by("name")` (core.py:147) before slicing with offset/limit, and `Action.name` is `CharField(null=True, blank=True)` with no DB uniqueness constraint. The reviewer's 'consider' rating is appropriate; keep as-is.
