_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** By design this PR writes through the `Action` model directly instead of `ActionSerializer`, replicating only the blank-name and name-uniqueness checks. Other validation the serializer performs is dropped: (1) `name` has no length check, but the model column is `CharField(max_length=400)`, so a name >400 chars from the LLM raises a database `DataError` at `action.save()` — unhandled, not a graceful `ActionToolError`; the REST path rejects it with a 400. This is an input-validation boundary gap for LLM-supplied data.
- **Suggestion:** Validate the boundary explicitly in the tool/core layer: check `len(name) <= 400` in `_check_name_available` (raising `ActionToolError` on overflow), and either validate the property-filter shape or wrap the property-driven `save()` so parse/compile failures become `ActionToolError` (retryable) instead of surfacing as raw DB/parse exceptions. This keeps parity with the validation `ActionSerializer` gives the REST path.
- **Validator:** Investigating both sub-claims against the actual code shows neither clears the bar. (1) Name length: the model field is `CharField(max_length=400)` and Postgres will raise `DataError` on a >400-char name at `save()`, and the REST serializer (an auto-generated ModelSerializer `name` field) does enforce that limit — so there is a real validation delta. Only exotic property types could raise a non-`BaseHogQLError`, and that same risk exists verbatim in the REST path — it is not a delta this PR introduces. The claimed 'the REST path rejects it with a 400' is inaccurate for properties. Given precision-over-recall, both parts are drop: one unreachable/minor, one resting on a mistaken premise. - **Problem:** The PR's stated reliability goal is that `list_actions` "is bounded by construction ... so discovery can never flood the agent's context." Reserve the full step/description rendering for `get_action`'s detailed view. Alternatively raise an `ActionToolError` for negative values so it surfaces as a retryable message the LLM can correct.

### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:147-150

**list_actions offset pagination uses a non-deterministic sort key, so rows can be skipped or duplicated across pages**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** `list_actions` paginates with `qs.order_by("name")` followed by an offset/limit slice, and the tool is explicitly designed around paging ("use `limit`/`offset` to page rather than trying to fetch everything"). But `Action.name` is `CharField(max_length=400, null=True, blank=True)` — nullable and non-unique (models/action.py:43). `order_by("name")` is therefore not a total order: any actions sharing a name (or the many actions with NULL/blank names) have an arbitrary, per-query relative order. With offset-based pagination this means the boundary between page N and page N+1 falls inside a group of tied rows whose order can differ between the two queries, so the same action can appear on consecutive pages or be skipped entirely when the agent walks pages via increasing `offset`. This defeats the paginated design and can make the agent believe an action doesn't exist (missed row) or double-count.
- **Suggestion:** Add a stable, unique tiebreaker to make the ordering total, e.g. `qs.order_by("name", "id")`. This guarantees a deterministic order across the separate queries the agent issues as it pages, eliminating skipped/duplicated rows. (Note this is distinct from the already-reported negative-offset crash — it concerns ordering determinism even for valid non-negative offsets.)
- **Validator:** The premise checks out: `Action.name` is `CharField(max_length=400, null=True, blank=True)` (products/actions/backend/models/action.py:43) — nullable and non-unique — and `list_actions` paginates with `qs.order_by("name")` then an offset/limit slice (core.py:147-150), with the tool explicitly designed for the agent to walk pages via increasing `offset`. `ORDER BY name` alone is not a total order: rows sharing a name, and the group of NULL/blank-named actions, have an arbitrary relative order that Postgres does not guarantee to be stable across the separate queries the agent issues per page (it can change under concurrent writes, autovacuum, or a different scan plan). When a tie group straddles a page boundary, the same action can reappear on consecutive pages or be skipped entirely — a silent correctness error in the tool's central paging use case, which can make the agent conclude an action doesn't exist (and e.g. create a duplicate) or double-count. This is a recognized real bug class (unstable offset pagination), not speculative paranoia or style, and the fix is the standard trivial one — append a unique tiebreaker, `order_by("name", "id")`. Reachability is genuine given unnamed/duplicate-named actions exist, though its practical frequency is bounded by needing a tie exactly at a page boundary plus an observable reorder, and most projects have few enough actions that paging is rare. On balance it clears the bar: a definite trigger, a real correctness consequence, and a one-word standard fix. should_fix is reasonable.
