- **Validator:** The premise checks out. `Action.name` is `CharField(max_length=400)` (products/actions/backend/models/action.py:43), and Django does NOT enforce `max_length` on `Model.save()` — it's only a validation-layer constraint that DRF's `ActionSerializer` applies at the REST boundary. `Model.save()` sends the raw value to Postgres, whose `varchar(400)` column rejects over-long strings with a `DataError`. The fix is trivial (a `Field(..., max_length=400)` bound, which also communicates the constraint to the LLM, or an explicit length check raising `ActionToolError`). ### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/core.py:147-150

**Non-deterministic pagination: order_by("name") over a non-unique, nullable column**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** list_actions paginates with offset/limit over `qs.order_by("name")`. Action.name is a non-unique, nullable CharField (products/actions/backend/models/action.py:43), so rows with equal (or NULL) names have no deterministic relative order in Postgres. Across paginated calls the tie order can change between queries, causing an action to be skipped or returned on two different pages as the agent walks offsets. `_check_name_available` only enforces uniqueness for future non-deleted writes; existing rows and REST-created actions can and do share names or be unnamed.
- **Suggestion:** Add a stable tiebreaker to make pagination deterministic, e.g. `qs.order_by("name", "id")`. The unique `id` guarantees a total order so offset-based paging can't skip or duplicate rows.
- **Validator:** This is a legitimate, well-understood correctness bug. `list_actions` does offset/limit pagination over `qs.order_by("name")` (core.py:147-150), and `Action.name` is a non-unique, nullable `CharField` (products/actions/backend/models/action.py:43). Offset pagination requires a total order to be stable across calls; with tied or NULL `name` values, Postgres has no guaranteed order among the tied rows, so the relative order can differ between the successive queries an agent issues while walking offsets — silently skipping an action or returning it on two pages. `_check_name_available` only blocks the tool from creating duplicate non-deleted names; it does not prevent existing rows, REST-created actions, or unnamed (NULL-name) actions from sharing a sort key, so ties genuinely occur. The trigger is realistic at the scale this tool targets (the tool description itself notes projects can have 'thousands of actions' and is built for paging), and the risk sharpens when the dataset changes between page fetches — likely in an agent flow that also creates/deletes actions. Concrete trigger (tied/NULL names + multi-page walk) and concrete consequence (skipped/duplicated rows in results shown to the LLM) are both nameable, so it clears the bar. should_fix is a reasonable priority for a silent correctness gap with a trivial fix.

- **Problem:** ListActionsTool.\_arun_impl returns the raw string from list_actions() directly into the LLM context. The design goal (stated in LIST_ACTIONS_DESCRIPTION and the PR body) is that this tool is 'bounded by construction' so 'discovery can never flood the agent's context'.
