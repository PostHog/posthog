The result is an action that silently matches far more events than the user asked for, then feeds insights/funnels with wrong data — and the model gets no signal to correct itself.
- **Suggestion:** Validate each step's `properties` eagerly in `create_action`/`update_action` before saving: for each property dict, attempt `Property(**prop)` (or run `property_to_expr(prop, team, strict=True)`) inside a try/except and raise `ActionToolError(f"Invalid property filter {prop!r}: {e}")` so the failure surfaces as a `MaxToolRetryableError` the LLM can fix. Alternatively, replace `properties: Optional[list[dict]]` in `ActionStepInput` with a typed pydantic filter model (key/value/operator/type with enums) so bad shapes are rejected at the args-schema layer.
- **Validator:** Confirmed the mechanism end-to-end. `ActionStepInput.properties` is `Optional[list[dict]]` (core.py:30-33) and `to_step_dict` only filters top-level step keys — the property dicts inside pass through untouched to `Action.steps`. At query/bytecode time `property_to_expr` (posthog/hogql/property.py) in its default non-strict mode catches `ValueError`/`TypeError` from `Property(**prop)` and returns `ast.Constant(value=1)` — the filter silently becomes match-everything, and since no HogQL error is raised, no `bytecode_error` is recorded and the tool reports success showing 'N property filter(s)'. Concrete trigger: an LLM emitting a filter dict missing `key`, or with a wrong shape (`Property.__init__` requires `key`; note it accepts `**kwargs`, so extra/misspelled keys like `values` are also silently ignored rather than erroring, producing a `value=None` filter). Concrete consequence: the created action over-matches, feeding wrong data into insights and funnels with no signal to the model or the user. The REST path has the same lax runtime validation, but its inputs come from the UI's filter builder, whereas free-formed LLM JSON is exactly the input this new surface makes common — so eager validation (typed pydantic filter model or `strict=True` check raising `ActionToolError`) is warranted here, not defensive paranoia. Real silent-data-corruption risk with a named trigger and consequence; keep at should_fix.
