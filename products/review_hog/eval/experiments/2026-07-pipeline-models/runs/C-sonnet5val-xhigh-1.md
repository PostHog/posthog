### [❌ dismissed] should_fix · bug — ee/hogai/tools/actions/core.py:181-189,192-205,208-226

**Check-then-act race on action name uniqueness has no DB backstop**  
_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** `_check_name_available` (lines 181-189) runs a plain `SELECT` to check for a name collision, and `create_action`/`update_action` call it before a separate `action.save()` a few lines later. `Action` has no `unique_together`/`UniqueConstraint` on `(team, name)` at the DB level (only a non-unique index on `team_id, -updated_at`), so this is a real TOCTOU race: two concurrent Max tool invocations (e.g. an agent retry firing while the first call is still in flight, or two chat sessions targeting the same team) can both pass `_check_name_available` and then both successfully `.save()`, producing duplicate-named actions with no exception raised by either call. `tool.py`'s wrappers only catch `ActionToolError`, so even if a unique constraint were added later, the resulting `IntegrityError` on `.save()` would propagate unhandled instead of being surfaced as a friendly retryable error.
- **Suggestion:** Wrap the check-and-save sequence in `transaction.atomic()` with `select_for_update()` scoping the name-collision query (e.g. lock existing rows for the team while checking), or add a DB-level `UniqueConstraint` on `(team, name)` filtered to `deleted=False` and catch `IntegrityError` around `action.save()`, converting it into an `ActionToolError` so the race fails safely and predictably instead of silently allowing duplicates.

Practical exposure is also narrow: each MaxTool invocation runs synchronously within one agent turn, so the only collision window is two independent chat sessions (or a chat session racing a REST client) hitting the same team+name within milliseconds — a rare timing window for a low-severity, non-corrupting outcome. The suggestion's second half (catch `IntegrityError` from a future constraint) is speculative since no such constraint exists today, so there's nothing to catch. Since the identical gap has existed indefinitely in the human-facing REST/UI path without a fix, holding this specific new Max tool to a stricter bar — rather than addressing it once at the model/serializer layer for both surfaces — is out of proportion to the actual (cosmetic, non-corrupting) impact. This reads as a pre-existing, already-tolerated characteristic rather than a new, user-affecting bug introduced by this change.
