### [✅ VALID] should_fix · bug — ee/hogai/tools/actions/tool.py:108-129

**UpdateActionTool never requires approval even when `steps` silently destroys all existing trigger conditions**  
_perspective: review-hog-blind-spots-general · directly-related: True_

- **Problem:** UpdateActionTool's own description (lines 53-56, `UPDATE_ACTION_DESCRIPTION`) states: "Providing `steps` REPLACES all existing steps, so include the full desired set." This is a fully destructive write — passing any `steps` list wipes every existing trigger condition for the action, with no diff/merge. Yet `UpdateActionTool` never overrides `is_dangerous_operation`/`format_dangerous_operation_preview` (it just inherits the `MaxTool` defaults at tool.py's base, which always return `False`/a generic message — see ee/hogai/tool.py lines 87-99), so this replacement runs immediately with no user confirmation, unlike `DeleteActionTool` in the very same file (lines 140-149), which is the only tool here gated behind the dangerous-operation approval flow. This is inconsistent with the established precedent for exactly this class of operation elsewhere in the codebase: `UpsertDashboardTool.is_dangerous_operation` (ee/hogai/tools/upsert*dashboard/tool.py) explicitly marks an update as dangerous specifically when it would \_replace/delete* existing tiles (`if isinstance(action, UpdateDashboardToolArgs) and action.insight_ids: ... return len(diff["deleted"]) > 0`), gating exactly this kind of "replace-and-lose-existing-config" mutation behind user approval. `UpdateActionTool` has the same replace-and-lose-existing-config shape (a set of steps entirely defines an action's matching behavior; anything downstream — insights, funnels — silently starts matching different events the moment `action.save()` runs) but ships with zero confirmation. An LLM that misremembers or hallucinates a subset of the current steps (very plausible, since the tool has no way to fetch-and-merge, only fetch-then-overwrite) can silently and irreversibly discard trigger conditions the user never asked to change, with no approval step to catch it before the write lands.
- **Suggestion:** Override `is_dangerous_operation`/`format_dangerous_operation_preview` on `UpdateActionTool`, mirroring `UpsertDashboardTool`'s pattern: treat the update as dangerous when `steps is not None` (i.e., a replacement is actually requested), and render a preview that shows the current steps that will be discarded and the new steps that will replace them, so the user can catch an unintended full replacement before it's applied. Example:

```python
async def is_dangerous_operation(self, action_id: int, steps: Optional[list[ActionStepInput]] = None, **kwargs) -> bool:
    return steps is not None

async def format_dangerous_operation_preview(self, action_id: int, steps: Optional[list[ActionStepInput]] = None, **kwargs) -> str:
    action = await _fetch_action(self, action_id)
    await self.check_object_access(action, "editor", action="edit")
    return (
        f"Replace all {len(action.steps)} existing step(s) on action #{action.id} "
        f"{action.name or '(unnamed)'} with {len(steps or [])} new step(s):\n{format_action_detail(action)}"
    )
```

- **Validator:** Confirmed against the code: `update_action` in ee/hogai/tools/actions/core.py unconditionally does `action.steps = [s.to_step_dict() for s in steps]` whenever `steps is not None` — a full overwrite with no diff/merge against the existing step list. `UpdateActionTool` (tool.py lines 108-129) never overrides `is_dangerous_operation`/`format_dangerous_operation_preview`, so it silently inherits `MaxTool`'s defaults (`False` / generic message), meaning this destructive replace executes with zero user confirmation. This is inconsistent with two things in the same PR/codebase: (1) `DeleteActionTool` right below it in the same file, which is explicitly gated behind `is_dangerous_operation` returning `True`; and (2) the established precedent in `UpsertDashboardTool.is_dangerous_operation` (ee/hogai/tools/upsert_dashboard/tool.py), which computes a diff and marks an update as dangerous specifically when it would delete existing tiles — i.e., the same team already solved 'replace-semantics update that can silently drop content' by gating it, and this PR doesn't apply that pattern to actions. I verified the test suite (ee/hogai/tools/actions/test/test_action_tools.py) has `test_delete_is_marked_dangerous` but no equivalent test for update — confirming the gap is real and untested, not just theoretical. Given the tool description itself documents 'steps REPLACES all existing steps' and the field is genuinely reachable any time an LLM calls update_action with a partial or misremembered steps list (very plausible since there's no fetch-and-merge path), this is a real, concrete data-loss/correctness risk with a direct, established fix pattern already in the codebase — not speculative paranoia.

### [❌ dismissed] should_fix · documentation — ee/hogai/tools/actions/core.py:30-33,47-48

**ActionStepInput.properties documents a restricted contract ('event'/'person') that isn't enforced, allowing 'hogql' property filters to reach the HogQL compiler**  
_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** The `properties` field on `ActionStepInput` is documented as accepting only `{'key', 'value', 'operator', 'type'}` where `type` is `'event'` or `'person'` (lines 30-33), but this is purely descriptive — there is no runtime enforcement. `to_step_dict()` (lines 47-48) uses `model_dump(exclude_none=True)` filtered only by top-level key name (`_STEP_FIELDS`); it never validates the _contents_ of the `properties` list, so any dict — including one with `"type": "hogql"` — is accepted verbatim and written into `Action.steps_json`.

That matters because `Action.refresh_bytecode()` (`products/actions/backend/models/action.py`) calls `action_to_expr(self)` -> `steps_to_expr()` -> `property_to_expr(step.properties, team)` (`posthog/hogql/property.py:1323-1324`). Inside `property_to_expr`, each raw dict is turned into a `Property(**property)` pydantic model, and `Property.type` is typed as `PropertyType`, which includes `"hogql"` as a valid member. When `property.type == "hogql"` (`posthog/hogql/property.py:810-812`), the code executes `parse_expr(property.key, cache_origin=CacheOrigin.USER)` — i.e. it compiles and runs the `key` string as arbitrary user-authored HogQL every time the action's bytecode is (re)computed, which happens on every `save()`.

This mirrors an existing gap in the REST `ActionStepJSONSerializer` (whose `properties` field is a plain `DictField()` with the same inaccurate help*text), so it isn't a net-new capability for a human API caller. But for this PR specifically, the field is the \_only* contract an LLM agent sees before deciding what to pass — an agent (potentially steered by content injected into the conversation) that trusts the docstring has no signal that supplying `type: 'hogql'` will cause its `key` value to be parsed and executed as HogQL against the team's ClickHouse data, and there is no validation layer here to catch or reject it before it's persisted and (re)compiled.

- **Suggestion:** Validate the `type` (and ideally `operator`) of each property filter dict against the documented allow-list (`'event'`, `'person'`) before accepting it in `to_step_dict()` / `ActionStepInput`, e.g. add a pydantic validator on `properties` that rejects (or strips) unsupported `type` values, or introduce a small `Literal["event", "person"]`-typed sub-model for property filter items instead of `list[dict]`. This makes the tool's actual behavior match its documented contract and closes the gap between what the LLM is told it can do and what the underlying model will actually execute.
- **Validator:** The premise treats `type: "hogql"` reaching `property_to_expr` as an unauthorized code-execution escape, but HogQL property filters are a fully supported, first-class filter type across the entire product — not a hidden or dangerous capability this PR newly unlocks. The same filter type is exposed through the standard property filter UI (`PropertyFilterType.HogQL` in `frontend/src/lib/components/PropertyFilters/utils.ts`), and any team member with edit access can already build an Action with a HogQL property filter via the existing Actions UI/REST API — `ActionStepJSONSerializer.properties` (products/actions/backend/api/action.py:75-80) is a plain `DictField()` with no `type` restriction either, so the REST path accepts and executes identical HogQL today. The `parse_expr(..., cache_origin=CacheOrigin.USER)` call in `property_to_expr` runs scoped, team-restricted HogQL (not arbitrary code) — it's the same execution path and same privilege level a human user already has when authoring a HogQL-filtered action, insight, or cohort. The MaxTool acts on behalf of the already-authenticated user within their own team, so no tenant-isolation or privilege-escalation boundary is crossed. The reviewer's own writeup concedes this 'mirrors an existing gap in the REST API... isn't a net-new capability' — which undercuts the security framing. The suggested fix (restrict `properties[].type` to `event`/`person` only) would actually reduce the tool's functional parity with the UI/REST API by blocking a legitimate, widely-used filter type, without closing any real vulnerability. The narrower, defensible core of this issue — that the docstring undersells the full accepted contract — is a minor documentation nit, not a security fix worth implementing as suggested.

Since these values are LLM-generated tool arguments rather than a human typing into a form, hitting an edge case like `offset=-1` or `limit=0` is a realistic failure mode, not a purely theoretical one.
- **Suggestion:** Add pydantic constraints (`Field(default=None, ge=1, le=MAX_LIST_LIMIT)` for `limit`, `Field(default=None, ge=0)` for `offset`), or validate/clamp explicitly at the top of `list_actions`, e.g. treat `limit is not None` (not truthiness) when applying the default, and raise `ActionToolError` for negative values so the agent gets a clear, retryable message instead of an unhandled `ValueError`.

The fix is a small, well-scoped pydantic constraint (`ge=1, le=MAX_LIST_LIMIT` / `ge=0`) consistent with existing patterns in the file, not speculative hardening — this passes the bar for a genuine correctness/reliability bug that will bite real LLM-driven usage.
