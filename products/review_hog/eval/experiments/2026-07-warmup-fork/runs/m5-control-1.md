**list_actions crashes on negative limit/offset and silently ignores an explicit limit=0**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** `ListActionsToolArgs.limit`/`offset` (core.py:56-60) have no lower-bound constraint, and `list_actions` (core.py:142-150) uses truthy-coalescing (`limit or DEFAULT_LIST_LIMIT`, `offset or 0`) instead of `is None` checks. This produces two distinct bugs: (1) an explicit `limit=0` is silently treated as `0 or 25 == 25` — the caller's clear intent ("return zero results") is discarded and 25 rows are returned instead; (2) any negative value for `limit` or `offset` flows straight into the Django queryset slice at line 150 (`qs[start : start + capped_limit]`). Django's `QuerySet.__getitem__` explicitly rejects negative slice bounds with `raise ValueError("Negative indexing is not supported.")` (confirmed in `django/db/models/query.py`), so e.g. `list_actions(team, None, -5, None)` or `list_actions(team, None, None, -1)` raises an uncaught `ValueError`. Unlike every other function in this module, `list_actions` never raises `ActionToolError`, so `ListActionsTool._arun_impl` (tool.py:69-73) has no try/except around the call — the exception propagates unhandled out of the tool invocation instead of being surfaced as a clean, LLM-retryable message the way every other validation failure in this module is (e.g. `ActionToolError` -> `MaxToolRetryableError`). An LLM producing a plausible-but-invalid negative parameter (not malicious, just a modeling mistake) breaks the tool call ungracefully.
- **Suggestion:** Add lower-bound validation. Either constrain the pydantic fields (`limit: Optional[int] = Field(default=None, ge=1, le=MAX_LIST_LIMIT, ...)`, `offset: Optional[int] = Field(default=None, ge=0, ...)`) so invalid values are rejected before reaching `list_actions`, or validate explicitly inside `list_actions` and raise `ActionToolError` for `limit is not None and limit < 1` / `offset is not None and offset < 0` — keeping it consistent with the rest of the module's user-facing error path. Also switch `offset or 0` / `limit or DEFAULT_LIST_LIMIT` to `offset if offset is not None else 0` / `limit if limit is not None else DEFAULT_LIST_LIMIT` so an explicit `0` isn't conflated with 'not provided'.
- **Validator:** Both claimed bugs are confirmed. (1) `limit or DEFAULT_LIST_LIMIT` turns an explicit limit=0 into 25 (0 is falsy), silently returning a default page instead of zero results — a silent wrong-result not caught by any handler, though an LLM requesting zero results is unlikely. (2) `start = offset or 0` and `min(limit or 25, 100)` preserve negative values (they are truthy), which flow into `qs[start : start + capped_limit]`; Django's QuerySet.**getitem** raises ValueError('Negative indexing is not supported.') on a negative slice start or stop. Because list_actions never raises ActionToolError, ListActionsTool.\_arun_impl has no try/except, so the ValueError escapes the tool. The framework's top-level `except Exception` in AgentToolsExecutable.arun catches it (Sentry capture + a generic 'internal error, do not retry' message), so there is no crash/500 or data loss — but it bypasses the clean, LLM-retryable MaxToolRetryableError path that every other user-fixable failure in this module routes through, so the LLM gets a worse, non-retryable hint plus spurious error-tracking noise. This is a real, reachable robustness/correctness defect directly in the PR's new code, with a plausible trigger (a negative pagination parameter is a natural LLM modeling mistake) and a trivial, idiomatic fix (pydantic ge=/le= bounds on the limit/offset fields plus `is None` checks instead of truthy coalescing). Bounding LLM-facing tool inputs at the schema level is standard practice, not defensive paranoia. Keeping at should_fix.

### [✅ VALID] consider · best_practice — ee/hogai/tools/actions/core.py:142-166

**list_actions pagination ordering has no deterministic tiebreaker**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** `list_actions` orders results with `qs.order_by("name")` (line 147) and then paginates via a plain offset/limit slice (line 150). `Action.name` has no DB-level uniqueness constraint (`products/actions/backend/models/action.py` — no `unique=True`, no `UniqueConstraint`, no `unique_together`); uniqueness is only checked application-side by `_check_name_available` (and equivalently by the REST `ActionSerializer.validate`), and that check is inherently racy (check-then-save, no DB constraint backing it), so two actions with an identical name can exist in a project. - **Validator:** The premise is verifiably correct. list_actions orders by `name` only (core.py:147) then paginates with a plain offset/limit slice (core.py:150), and Action.name has no DB uniqueness constraint (CharField with no unique=True; the app-side_check_name_available query exists precisely because there is no DB constraint, and it is racy and does not cover NULL/blank names from other creation paths). So ties (duplicate or NULL names) can exist, and without a unique tiebreaker Postgres does not guarantee a stable order among equal-key rows across separate LIMIT/OFFSET queries — successive paginated calls (which the tool's docstring instructs the LLM to make) can non-deterministically duplicate or skip a tied action. This is a genuine, well-established pagination-stability convention, and the fix (order_by('name', 'id')) is a one-token, zero-risk change in the PR's new code. Impact is minor and self-correcting: the trigger is narrow (tied/NULL names AND active pagination AND a tie at a page boundary) and the worst case is a discovery/listing tool briefly showing a dup or miss across pages — no data loss, corruption, or security impact. It is a real, correct, directly-related best-practice finding, not speculative or wrong, so it stays on record. The reviewer's `consider` priority is appropriate — it keeps the finding recorded without surfacing it prominently, which fits a real-but-minor issue.
