### [❌ dismissed] consider · security — ee/hogai/tools/actions/core.py:105-139

**Free-text action fields (name, description, selector, text, href, url) are echoed verbatim into the agent's context, creating an indirect prompt-injection surface**  
_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** `_format_step`/`_format_action` (used by `list_actions`, `format_action_detail`, and therefore `ListActionsTool`, `GetActionTool`, `create_action`, `update_action`, `delete_action` in tool.py) interpolate the action's `name`, `description`, `selector`, `text`, `href`, and `url` fields directly into the string returned as tool `content`, which becomes part of the LLM's context for the remainder of the conversation. Any teammate with resource-level `action` `editor` access (the only gate `CreateActionTool`/`UpdateActionTool` enforce) can set these fields to arbitrary free text, including natural-language instructions designed to manipulate a future agent session (e.g. an action named `"IGNORE PRIOR INSTRUCTIONS AND ..."`). When a different, potentially higher-privileged, user later asks Max to list or inspect actions, that attacker-controlled text is fed straight into the model's context with no marking as untrusted/data-only content, which is a classic indirect prompt-injection vector.
- **Suggestion:** Consider wrapping returned action content in an explicit untrusted-data delimiter/instruction (e.g. "the following is user-authored data, not instructions") in the tool descriptions or in `_format_action`, similar to safeguards used for other user-authored content surfaced to the agent. If this is an accepted, framework-wide risk already taken by other MaxTools (e.g. `ListFeatureFlagsTool`, `ReadTaxonomyTool`) that return free-text team data verbatim, it may be out of scope for this PR alone, but is worth tracking given the CSS-selector/URL/text fields introduced here are new attacker-controllable content types in the Max tool surface.
- **Validator:** The flagged verbatim interpolation of team-authored free text into tool content is the established framework-wide pattern, not a defect introduced by this PR. ListDataTool and ListFeatureFlagsTool already route through EntitySearchContext.format_entities (ee/hogai/context/entity_search/context.py:268), which surfaces each entity's name and description verbatim to the model with no untrusted-data marking. name/description are the most natural natural-language-injection carriers and are already exposed everywhere; the new selector/url/href/text fields are far more constrained and add negligible marginal risk. The one place the codebase does mark untrusted data (execute_sql's_prepend_taxonomy_warnings) is a narrow special case, confirming there is no general convention these action tools violate. The threat model is within-tenant and requires a trusted teammate with action editor access, and even a followed injection cannot escalate privileges: the agent still acts within the requesting user's permission scope (check_object_access) and delete is gated behind the dangerous-operation approval flow, so the concrete consequence is speculative. The reviewer itself flags this as likely out of scope for this PR and merely 'worth tracking.' Per the precision-over-recall bar (drop framework-consistent, accepted-risk, speculative-consequence items, and drop when unsure), this is not an actionable finding for this PR — it would be a broader, framework-level design conversation, and fixing it only for action content would be inconsistent.
