### [❌ dismissed] should_fix · documentation — ee/hogai/tools/actions/tool.py:42-42

**list_actions description tells the agent to use read_taxonomy to view an action's steps, but that tool can't do this**  
_perspective: review-hog-perspective-logic-correctness · directly-related: True_

- **Problem:** LIST_ACTIONS_DESCRIPTION's closing sentence says: "Use this to discover an action's ID before reading its properties (read_taxonomy), updating, or deleting it." This directs the agent to call `read_taxonomy` to read an action's properties/step definition. However, ReadTaxonomyTool only supports query kinds `events`, `event_properties`, `event_property_values`, `entity_properties`, `entity_property_values`, `action_properties`, `action_property_values` (ee/hogai/tools/read_taxonomy/core.py). The `action_properties`/`action_property_values` kinds run an EventTaxonomyQuery against events matching the action to surface event property names/values — they do not return the action's own step/trigger-condition definition (selector, URL match, event name, etc.). The tool that actually returns that — `GetActionTool` (name `get_action`) — is defined in this same file with the description "Get a single action by ID, including its full step (trigger condition) definition." UPDATE_ACTION_DESCRIPTION (line 55) correctly points to `get_action` for the same purpose ("Look up the current definition with get_action first"), confirming the `read_taxonomy` reference on line 42 is a mistake, not an intentional alternate path. Since these description strings are the LLM's only guide for tool selection, this could lead the agent to call `read_taxonomy` (with action_properties/action_property_values) expecting to see an action's steps, get back unrelated event-property statistics, and either loop retrying the wrong tool or draw wrong conclusions about the action's configuration before an update/delete call.
- **Suggestion:** Change the parenthetical on line 42 from `(read_taxonomy)` to `(get_action)`, matching the reference already used in UPDATE_ACTION_DESCRIPTION: "Use this to discover an action's ID before reading its properties (get_action), updating, or deleting it."
- **Validator:** The reviewer's premise is inaccurate. Line 42 says the agent should discover an action's ID 'before reading its properties (read_taxonomy)'. read_taxonomy genuinely reads an action's properties: its ReadActionProperties kind (action_properties) is documented as 'Returns the properties list for a provided action' and routes to toolkit.retrieve_event_or_action_properties(action_id) in ee/hogai/tools/read_taxonomy/core.py. The description says 'reading its properties', not 'reading its steps' or 'its definition' — so referencing read_taxonomy for that is correct, not a mistake. The reviewer conflated two distinct, both-valid flows: read_taxonomy returns an action's associated event properties, while get_action returns the action's step/trigger-condition definition. UPDATE_ACTION_DESCRIPTION correctly points to get_action for looking up the definition before editing steps; that does not imply line 42's reference to read_taxonomy for 'reading properties' is wrong — they serve different purposes. There is no contradiction to fix. Additionally, get_action's own description clearly states it returns 'the full step (trigger condition) definition', so an agent needing the step definition is guided there by the toolset regardless of this parenthetical. What remains is at most a wording-clarity preference with negligible real impact on tool selection, which falls under documentation taste rather than a correctness or contract issue worth surfacing.

### [✅ VALID] should_fix (validator→consider) · best_practice — ee/hogai/tools/actions/tool.py:108-129

**UpdateActionTool's destructive full-replace of `steps` isn't gated by the dangerous-operation approval flow**  
_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** UpdateActionTool's own description (UPDATE_ACTION_DESCRIPTION, lines 53-56) states that providing `steps` "REPLACES all existing steps" — an LLM-driven update that omits or truncates the desired set silently discards the action's other trigger conditions, with no way for the framework to intervene before the write happens. This class does not override `is_dangerous_operation` (base class default in ee/hogai/tool.py:87-94 returns False), so `update_action` runs immediately whenever the model calls the tool — unlike `DeleteActionTool` in the same file (lines 140-141), which explicitly returns `True` and requires human approval before executing. The codebase already has a precedent for exactly this failure mode: `UpsertDashboardTool.is_dangerous_operation` (ee/hogai/tools/upsert_dashboard/tool.py:109-115) computes a diff and flags the update as dangerous specifically "Update operations that delete existing insights" before letting an LLM-driven edit proceed. `UpdateActionTool` has the analogous risk (an LLM update silently dropping existing steps) but ships without the safeguard, relying solely on prompt text to steer the model into calling `get_action` first — a documentation-only mitigation that doesn't hold up against a model that misreads or ignores the instruction.
- **Suggestion:** Fetch the action's current `steps` (already available via `_fetch_action`) and diff it against the incoming `steps` argument inside an `is_dangerous_operation` override; return True when steps are being provided and would remove/change existing entries (mirroring `UpsertDashboardTool.is_dangerous_operation`'s diff-then-decide pattern), and surface the removed steps in `format_dangerous_operation_preview` so the user can see what's about to be dropped before approving.
- **Validator:** The reviewer correctly observes that UpdateActionTool does not override is_dangerous_operation (tool.py:108-129), so an LLM-driven update that supplies an incomplete `steps` list will fully overwrite action.steps immediately (core.py:222-223), with no approval step — unlike DeleteActionTool in the same file and unlike UpsertDashboardTool, which conditionally gates updates that delete existing insight tiles (upsert_dashboard/tool.py:109-116). That is a real pattern with a genuine codebase precedent, so it isn't noise. However, the severity is lower than should_fix: (1) the operation is recoverable rather than permanent loss — update_action goes through Action.save() under_acting_user, which emits activity-log entries, the action is not deleted, and the tool echoes the full new step set back to the user; (2) the delete-vs-update distinction the author drew is defensible — delete removes the whole action from every insight/funnel, while update mutates one field of one object; (3) the UpsertDashboardTool gate is itself conditional and covers whole saved insights, a larger unit of user work than a single action's steps; and (4) the replace semantics are explicitly documented in both UPDATE_ACTION_DESCRIPTION and the `steps` field description, with the model instructed to call get_action first, so the harmful path requires the model to ignore explicit instructions AND the user to overlook the echoed result. Given recoverability, the deliberate design line, and the existing prompt guidance, this is a real-but-minor consistency/reliability concern rather than a blocking defect. Keeping it on record at `consider` reflects that without over-surfacing it.

### [❌ dismissed] consider · best_practice — ee/hogai/tools/actions/tool.py:90-105

**CreateActionTool has no bound on runaway action creation within a single tool surface**  
_perspective: review-hog-perspective-performance-reliability · directly-related: True_

- **Problem:** `ListActionsTool` is explicitly designed with a hard cap (`MAX_LIST_LIMIT = 100` in core.py) specifically "so the tool can never blow up the agent's context window," but `CreateActionTool._arun_impl` has no equivalent safeguard against being called repeatedly (e.g. an LLM looping while trying to construct a multi-step action, or misinterpreting a batch request as N separate creates). The only backstop is `check_count_limit` inside `create_action` (core.py), which per its own comment "is notification-only — check_count_limit never raises or blocks." So there is no hard limit anywhere in the create path that can stop a single conversation from creating an unbounded number of actions in a tight loop, unlike the read path's deliberate ceiling.
- **Suggestion:** Consider adding a lightweight guard in the tool layer (e.g. reject/require confirmation once the team's action count crosses a high-water mark, or cap creates per conversation turn) so a misbehaving tool-call loop can't create an unbounded number of rows before a human notices — consistent with the bounded-by-construction philosophy already applied to `list_actions`.
- **Validator:** This is a speculative defensive safeguard against an unlikely LLM-loop scenario, not a concrete defect. The analogy to ListActionsTool's MAX_LIST_LIMIT is flawed: that cap prevents a single response from blowing the agent's context window — an inherent, guaranteed problem with no downside to capping — whereas 'runaway creation' is a different, speculative failure mode. For unbounded creation to occur, an LLM would have to loop while supplying a unique name on every call (create_action calls_check_name_available first, which rejects duplicate non-deleted names via ActionToolError -> MaxToolRetryableError, core.py:181-189), the loop would have to defeat the agent framework's per-turn tool-call iteration limits, and the resulting rows are soft-deletable and fully recoverable. That is a compounding 'what if the model misbehaves' chain, not a path real inputs reliably hit. Moreover, check_count_limit is intentionally notification-only to match the REST/UI create path; adding a hard block or confirmation gate only in the Max tool would diverge from the product's established non-blocking action-creation semantics — a product-policy change, not a bug fix. The suggestion is explicitly framed as 'consider adding a lightweight guard,' with no named trigger-and-consequence that concrete inputs will hit. Under precision-over-recall, and the criteria's exclusions for speculative what-ifs, defensive-coding paranoia, and add-a-safeguard overengineering, this should be dropped.

### [❌ dismissed] should_fix · security — ee/hogai/tools/actions/core.py:94-102

**`_acting_user` only swaps `user` on `activity_storage`, leaving `was_impersonated`/`client`/`ip_address` at whatever was previously set on the same thread/task context**  
_perspective: review-hog-blind-spots-general · directly-related: True_

- **Problem:** `_acting_user` is meant to stand in for the request middleware that normally populates `activity_storage` before a save (the docstring says exactly this: "no request middleware here"). But it only saves/restores `get_user()`/`set_user()`. The activity-log signal handlers for `Action` (fired from `ModelActivityMixin.save()` on every `create_action`/`update_action`/`delete_action` call in this file) also read `activity_storage.get_client()`, `get_ip_address()` (as fallback defaults inside `log_activity`) and, notably, `get_was_impersonated()` (read unconditionally by the signal handler, not defaulted) to build the `ActivityLog` row. In the normal request path, `ActivityLoggingMiddleware` sets all four of these per request and unconditionally clears all four (`clear_all()`) in a `finally` at the end of the request. `_acting_user` mimics only 1 of those 4 fields and never clears the other 3. Since `ActivityLoggingStorage` is built on `asgiref.local.Local` (thread/async-task-scoped, not request-scoped), if this Max-tool code path executes on a thread/task that previously carried impersonation or client/IP state from unrelated code (and that state was never cleared, e.g. because it wasn't inside a full middleware-wrapped request cycle), the resulting `ActivityLog` entry for the action create/update/delete can silently inherit a stale `was_impersonated` flag or `client`/`ip_address` value that has nothing to do with the actual Max-tool caller. This directly undermines the audit-trail accuracy that activity logging exists to provide — `was_impersonated` in particular is a security-relevant field (it's how staff-impersonation actions are distinguished from a real user's own actions), and this code path doesn't guarantee it's correct.
- **Suggestion:** Have `_acting_user` snapshot and restore (or explicitly reset for the duration of the block) all four fields the signal handlers consume, not just `user` — e.g. also `set_was_impersonated(False)`, and either pass through explicit `client`/`ip_address` or reset them to `None`, then restore the previous values in `finally` alongside `user`. This makes the context manager symmetric with what `ActivityLoggingMiddleware` actually guarantees, instead of assuming the other three fields happen to already be in a safe state.
- **Validator:** The mechanism described is accurate (the model_activity signal reads was_impersonated from activity_storage, and log_activity falls back to activity_storage for client/ip_address, while \_acting_user only swaps user), but the described failure is unreachable given how this code actually runs. The Max action tools execute inside a Temporal activity (process_chat_agent_activity -> ChatAgentRunner -> database_sync_to_async(create_action)) in a dedicated Temporal worker process, not behind ActivityLoggingMiddleware. A codebase-wide search confirms the ONLY writers of was_impersonated/client/ip_address on activity_storage are ActivityLoggingMiddleware and ImpersonatedContext, both of which live exclusively in the Django web process (middleware + two DRF viewsets) and both clear their state. Nothing in the Temporal worker process ever writes those three fields, so get_was_impersonated() always returns its default False and get_client()/get_ip_address() always return None in this path — there is no stale value to inherit. asgiref.local.Local is additionally context-isolated per async task, so concurrent conversations cannot cross-pollute. The suggested fix (reset was_impersonated=False, client/ip=None) would therefore be a no-op in the real execution context; it only guards a polluted state that cannot occur here. This is a speculative what-if / defensive-symmetry concern against an unreachable condition, which the precision-over-recall bar says to drop. (Note: a genuine, opposite-direction gap exists — a staff-impersonated Max session would currently log was_impersonated=False because inputs.is_impersonated is never wired into activity_storage — but that is under-reporting, not the stale over-reporting this issue flags, and the proposed fix would not address it.)

### [❌ dismissed] should_fix · bug — ee/hogai/tools/actions/core.py:30-33,192-226

**Unvalidated `properties` filter dicts can crash `action.save()` with an uncaught `ValueError` during bytecode compilation**  
_perspective: review-hog-perspective-contracts-security · directly-related: True_

- **Problem:** `ActionStepInput.properties` (lines 30-33) is typed `Optional[list[dict]]` with no shape validation — any dict passed by the LLM flows unchanged through `to_step_dict()` into `Action.steps`, and ultimately into `Property(**property)` during bytecode compilation (`posthog/hogql/property.py` / `posthog/models/property/property.py`). `Property.__init__` raises a plain `ValueError` for an invalid `type` or a missing `value` on non-`is_set`/`hogql` operators — this is not a `BaseHogQLError`, so `Action.refresh_bytecode()`'s `except BaseHogQLError` handler does not catch it, and it propagates uncaught out of `action.save()` in both `create_action` (lines 192-205) and `update_action` (lines 208-226). Because this exception is a bare `ValueError` (same base class as this module's own `ActionToolError`, but not an instance of it), it bypasses the module's `ActionToolError` handling entirely and is only caught by the agent graph's generic exception fallback, producing a non-retryable "internal error" message and a Sentry capture for what is a normal user-input mistake the LLM could otherwise be asked to correct. This is a materially larger exposure than the equivalent REST path: REST at least enforces `ListField(child=DictField())` at the DRF layer before this point, whereas the direct-ORM tool path (by design, to bypass `ActionSerializer`) has no equivalent shallow guard at all.
- **Suggestion:** Either validate the shape of each `properties` entry against the fields `Property(...)` actually expects (key/value/operator/type, with `type` restricted to the known literal set) inside `ActionStepInput`/`to_step_dict()` so bad input fails fast with a pydantic error, or wrap the `action.save()` calls in `create_action`/`update_action` with a `try/except ValueError` that re-raises as `ActionToolError` with a descriptive message — consistent with how blank/duplicate names are already surfaced as retryable errors in this module.
- **Validator:** The core premise is factually wrong. The issue claims a bare ValueError from Property.**init** propagates uncaught out of action.save() during bytecode compilation. But the actual path — refresh_bytecode() -> action_to_expr() -> steps_to_expr() -> property_to_expr(step.properties, team) (posthog/hogql/property.py:1324) — invokes property_to_expr with strict=False (the default). For each property dict, property_to_expr wraps Property(**property) in `try/except ValueError` AND `except TypeError` (property.py:739-751), and because strict=False it returns ast.Constant(value=1) instead of re-raising. So the exact ValueError raised by Property.**init\*\* for an invalid `type`/missing `value` (property.py:298/303/307), and TypeError from unexpected keys, are caught inside property_to_expr and never reach refresh_bytecode's BaseHogQLError handler, let alone escape action.save(). Malformed property dicts are silently treated as no-op filters; bytecode compilation completes normally. The author overlooked this except block. Secondary claims are also incorrect: the tool path does have a shallow guard (pydantic validates ActionStepInput.properties as Optional[list[dict]], equivalent to REST's ListField(child=DictField())), and neither the tool nor REST validates the deeper Property shape — both rely on the same graceful non-strict handling in property_to_expr, so there is no larger exposure on the tool path. The suggested try/except ValueError or shape validation defends against a crash that cannot occur here, making this unreproducible.

### [✅ VALID] must_fix · security — ee/hogai/tools/actions/tool.py:61-73


```python
# tool.py
async def _arun_impl(self, search=None, limit=None, offset=None) -> tuple[str, None]:
```
