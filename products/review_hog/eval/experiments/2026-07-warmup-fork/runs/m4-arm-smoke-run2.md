### [✅ VALID] must_fix · bug — ee/hogai/tools/actions/core.py:105-139

**get_action/create_action/update_action never surface property-filter contents, contradicting the tool's "full step definition" contract and breaking the documented safe-update workflow**  
_perspective: review-hog-blind-spots-general · directly-related: True_

- **Problem:** `_format_step` (lines 105-121) is the ONLY rendering path for a step's contents, and for property filters it renders nothing but a count: `if step.properties: parts.append(f"{len(step.properties)} property filter(s)")` (line 119-120) — the actual `key`/`value`/`operator`/`type` of each filter is never emitted. Critically, `_format_action`'s `detailed=True` branch (lines 130-131, used by `get_action`/`create_action`/`update_action` via `format_action_detail`) calls the exact same `_format_step` per step as the non-detailed listing branch (line 134) — there is no additional richness gained by `detailed=True`; it only changes whether steps are joined on one line or given their own line. So no tool in this module can ever show what a property filter actually matches. This directly contradicts `GET_ACTION_DESCRIPTION` in tool.py ("Get a single action by ID, including its full step (trigger condition) definition") for any action using property filters, and it breaks the safe partial-update flow the tool itself recommends: `UpdateActionToolArgs.steps`'s docstring says steps REPLACE all existing steps and instructs callers to "Look up the current definition with get_action first if you only want to tweak part of it" — but get_action cannot recover the current property-filter values, so an agent trying to preserve existing property filters while tweaking one step has no way to see what to preserve and risks silently dropping or fabricating filter content on the next update_action call. This is untested: test_action_tools.py never creates a step with `properties` set, so the gap has no regression coverage.
- **Suggestion:** Render each property filter's key/operator/value (and type) in `_format_step`, at least when `detailed=True`, e.g.:

```python
if step.properties:
    if detailed:
        filters = "; ".join(
            f"{p.get('key')} {p.get('operator', 'exact')} {p.get('value')!r} ({p.get('type', 'event')})"
            for p in step.properties
        )
        parts.append(f"properties: {filters}")
    else:
        parts.append(f"{len(step.properties)} property filter(s)")
```

This requires threading `detailed` into `_format_step` (or duplicating a detailed variant), so `get_action`/`create_action`/`update_action` actually deliver on the "full step definition" they promise and callers can safely round-trip existing property filters through update_action.

- **Validator:** Confirmed in the code. \_format_step (core.py:105-121) renders event/url/selector/tag_name/text/href with their full values but collapses property filters to just a count ('N property filter(s)') — the key/value/operator/type are never emitted.\_format_action's detailed=True branch (core.py:130-134) calls the exact same_format_step, so no code path — including get_action, create_action, and update_action's outputs — ever shows what a property filter matches. This directly contradicts GET_ACTION_DESCRIPTION's promise of 'its full step (trigger condition) definition' (tool.py:45) and breaks the safe-update workflow UPDATE_ACTION_DESCRIPTION itself recommends: steps REPLACES all existing steps, and callers are told to 'Look up the current definition with get_action first if you only want to tweak part of it' (tool.py:53-56). Because get_action cannot surface property-filter contents, an agent tweaking one step while preserving others must reconstruct the full steps set blind — silently dropping or fabricating filter contents on the next update_action. The loss is invisible: update_action's own confirmation (core.py:226) also uses detailed=True and hides it. Property filters on action steps are a common PostHog data shape, not a speculative edge case, so the trigger (action with property filters) and consequence (data loss on round-trip) are both concrete and realistic. This is not overengineering, style, or defensive paranoia — it's a real correctness/data-loss gap that undermines the tool's stated contract, and it has no regression coverage (no test sets properties on a step). must_fix is appropriate given the silent data-loss potential via the documented workflow.
