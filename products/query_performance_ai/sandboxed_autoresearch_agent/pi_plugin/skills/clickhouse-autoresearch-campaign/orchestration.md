# ClickHouse Query Optimization Orchestration Contract

This document is the durable reasoning contract for the ClickHouse optimization campaign.

## Mission

Find generalizable performance improvements using one slow ClickHouse query as the investigation target.

The target query is generated by PostHog's query runners. Optimizations that only help this exact query (e.g., adding a hardcoded event filter) have little value. Focus on improvements that would benefit many queries with different parameters.

Run a disciplined campaign that:

- builds a baseline dossier
- maintains a dynamic backlog of lanes and hypotheses
- runs experiments, classifying each finding as generalizable or query-specific
- compares performance against the baseline
- compares results against the saved baseline result set
- keeps validated improvements
- discards regressions
- reflects after every experiment
- maintains `out-of-scope-suggestions.md` — recommendations for schema-level changes the agent suspects would help but cannot apply in the sandbox (materialized columns, skip indexes, projections, table-engine swaps)
- decides when to continue, pivot, integrate, repair, review, or close the campaign

## Core model

- **Campaign**: one investigation for one target query on one git branch.
- **Lane**: one optimization direction tied to a suspected bottleneck or strategy.
- **Hypothesis**: one concrete, testable idea within a lane.
- **Experiment**: one executed hypothesis with benchmark and correctness evaluation.

## Fundamental rules

- One campaign corresponds to one git branch.
- The branch maintains one canonical best query candidate.
- Kept wins should normally accumulate on top of the current best candidate.
- Hypotheses come from the agent's own reasoning over the baseline dossier; treat agent-generated hypotheses as the only input.
- The backlog is dynamic and must be updated after every experiment.
- Do not overfit to the benchmark or cheat the workload.
- Semantics must be preserved relative to the saved baseline result set unless explicitly exploring a bounded repair of a promising incorrect candidate.
- Lane review may pause or close a lane, but only campaign review may close the campaign.
- Write explicit rationale summaries, not hidden chain-of-thought.

## Optimization categories

Prioritize improvements by generalizability:

### 1. Schema-level (highest priority, cannot test directly)

Materialized columns, skip indexes, projections, table engine changes. The agent cannot ALTER TABLE on the test cluster — these are out of scope for the sandbox. Instead:

- Identify the opportunity from EXPLAIN plans, row counts, and query patterns
- Record the suggestion in `out-of-scope-suggestions.md` with evidence and estimated impact (this file is the agent's persistent output for things it suspects would help but cannot validate here)
- Describe which query patterns would benefit (not just this one query)

### 2. Query generation patterns (can test by rewriting)

Changes that should be applied in PostHog's query runner code:

- Timestamp handling: `toDateTime(literal, tz)` instead of `toTimeZone(timestamp, tz)` per row
- JSON function choice: `JSONExtractString` over `replaceRegexpAll(JSONExtractRaw(...))`
- Predicate ordering: place high-selectivity filters that use the primary key first
- Unnecessary casts, redundant expressions, suboptimal JOIN strategies

Test by rewriting the query. Kept rewrites live in `query/best.sql`; if the rewrite encodes a reusable pattern, note it in `autoresearch.md` alongside the benchmark evidence.

### 3. Query-specific (lowest priority)

Hardcoded filters, literal substitutions, or structural changes that only help this exact query. Only pursue if budget remains after the above categories are explored.

### Out-of-scope suggestions file

`out-of-scope-suggestions.md` is the agent's persistent record of schema-level changes it suspects would help but cannot apply in the sandbox: materialized columns, skip indexes, projections, table-engine swaps. The orchestrator harvests this file at the end of the campaign and surfaces it as a deliverable, so each entry must be self-contained and actionable. Required fields:

- **What to change** (e.g., "materialize `JSONExtractString(properties, '$email')` as a column")
- **Evidence**: EXPLAIN plan excerpts, row counts, benchmark measurements
- **Expected impact**: estimated improvement and which queries would benefit
- **Applicability**: how broadly this applies (one query kind, many query kinds, all queries)

## Campaign initialization

Before active experimentation:

1. Identify the target query.
2. Create or verify the campaign branch.
3. Build the baseline dossier.
4. Extract initial lanes from:
   - query structure
   - profiling / EXPLAIN / ANALYZE evidence
   - schema / table characteristics
5. Create initial hypotheses.
6. Initialize the experiment session and establish the baseline.

## Timeout queries and range narrowing

Some target queries time out on the test cluster with their original time range. When this happens, narrow the range until the query is runnable, optimize there, and periodically re-test against the original range.

### When to activate

Enter range narrowing when the baseline capture fails with a timeout or exceeds `max_execution_time`.

### Narrowing phase

Reduce the time range in the query (halve it repeatedly: 30d → 14d → 7d → 3d → 1d) until the query completes. The target is a working range that is still meaningfully slow (1–10s), not trivially fast (sub-100ms). If the query becomes trivially fast at a given range, step back to the previous wider range and try intermediate values.

### File conventions

- `query/original.sql` — the full-range query, never modified
- `query/narrowed.sql` — the reduced-range working copy
- `query/current.sql` — the candidate under test (starts as a copy of the narrowed query)
- `query/best.sql` — the best kept candidate so far

### Working baseline

Capture the baseline against the narrowed query. The correctness reference is the narrowed query's result set — results will differ from the full range and that is expected.

Record the narrowing state in `state.json`:

```json
{
  "narrowed": true,
  "original_range": "30d",
  "working_range": "3d"
}
```

### Escalation checks

After every `keep` result, test whether the current best candidate is viable at the original time range:

1. Apply the same optimizations from `query/best.sql` to a copy using the original time range
2. Run it with a generous but bounded timeout (e.g., `max_execution_time=60`)
3. Record whether it completed or timed out, and if it completed, the latency

Log escalation results in `state.json` under an `escalation` key:

```json
{
  "escalation": [
    { "after_run": 5, "completed": false },
    { "after_run": 12, "completed": true, "latency_ms": 8400 }
  ]
}
```

### Graduation

When an escalation check succeeds (original range completes within a reasonable time):

1. Switch the campaign to the full range
2. Re-capture the baseline with the full-range query
3. Update the correctness reference to the full-range result set
4. Continue optimizing from the current best candidate

### Correctness rule during narrowing

During the narrowed phase, correctness is validated against the narrowed baseline. After graduation, correctness is validated against the full-range baseline.

## Baseline dossier requirements

The baseline dossier must include:

- original query text
- relevant schema / table definitions
- relevant EXPLAIN / ANALYZE / profiling artifacts
- baseline performance metrics
- saved baseline result set from actually executing the query
- assumptions required to reproduce the experiment

The saved baseline result set is the semantic reference for future correctness checks.

## Lane and hypothesis management

Treat lanes and hypotheses as a living structure.

At all times:

- maintain lane status: active, paused, repair, exhausted, or closed
- maintain hypotheses sourced from:
  - agent-generated evidence
  - reasoning over the baseline dossier and EXPLAIN output
- merge duplicates
- split overly broad hypotheses
- create new lanes when new bottlenecks appear
- create integration hypotheses when validated wins from different lanes may combine
- explicitly record when no new hypothesis can be extracted

When choosing the next experiment, prefer:

- high expected upside
- strong evidence support
- manageable semantic risk
- compatibility with already-kept improvements
- high information value

## Experiment protocol

For each experiment:

1. Select the next lane and hypothesis.
2. Record a concise rationale:
   - suspected bottleneck
   - supporting evidence
   - intended change
   - expected metric movement
   - semantic risks
3. Modify the current candidate query.
4. Run the benchmark and capture performance artifacts.
5. Run correctness comparison against the saved baseline result set.
6. Adjudicate the outcome:
   - improved and correct -> keep
   - not improved and correct -> discard
   - improved but incorrect -> evaluate repair-worthiness
   - crash / invalid -> discard unless trivially fixable
7. Reflect on the result.
8. Update the lane and hypothesis backlog.
9. Decide the next action:
   - continue current lane
   - narrow current lane
   - switch lanes
   - create a new lane
   - open an integration hypothesis
   - open a repair hypothesis or repair lane
   - trigger lane review
   - trigger campaign review

## Mandatory post-experiment reflection

After every experiment, explicitly assess:

- what prediction was confirmed
- what prediction was wrong
- what unexpected signal appeared
- whether the measured effect happened for the expected reason
- whether correctness failure, if any, looks trivial, repairable, or fundamental
- whether the result strengthens, weakens, splits, or merges any hypotheses
- whether the result suggests a new hypothesis
- whether the result suggests a new lane
- whether no meaningful new hypothesis can be extracted
- whether the optimization is generalizable (schema-level or query-generation pattern) or query-specific
- if it implies a schema change the sandbox cannot apply: record it in `out-of-scope-suggestions.md`

Reflection is mandatory even when the result is inconclusive.

## Lane-level stagnation / exhaustion review

Lane review is tactical.

Trigger lane review when the current lane shows signs of exhaustion, such as:

- no significant improvement from this lane for the configured lane stagnation window
- no meaningful new hypotheses extracted from this lane for the configured lane stagnation window
- repeated similar plans or metrics with little new signal
- only trivial variants remain
- repair attempts inside this lane are not converging

Lane review may conclude:

- continue the lane
- narrow the lane
- split the lane
- merge it with another lane
- pause / park the lane
- close the lane as exhausted

Lane review must not end the campaign.

## Campaign-level stagnation / exhaustion review

Campaign review is strategic.

Trigger campaign review when:

- the active lane is exhausted and no strong lane is ready
- the campaign has seen no significant improvement for the configured campaign stagnation window
- recent reflection is not generating meaningful new hypotheses or new lanes
- remaining lanes are low-confidence, repetitive, or low expected value
- repair paths are exhausted
- iteration budget is approaching exhaustion

Campaign review may conclude:

- continue with another lane
- continue with narrowed scope
- continue with an integration direction
- continue with a repair direction
- close the campaign

A campaign should continue only while it is still producing either:

- meaningful gains, or
- genuinely new learning with realistic upside

## Combining results from different lanes

Combining good results across lanes is expected.

Default behavior:

- explore new hypotheses on top of the current best kept candidate so wins accumulate naturally

If strong results were discovered partly in isolation:

- explicitly create an integration hypothesis to test them together

When evaluating lane combinations, classify the relationship:

- additive
- enabling
- conflicting
- redundant

Do not exhaustively combine everything. Only test combinations with credible synergy or transfer value.

A lane result is not fully banked until it has either:

- been integrated into the current best candidate, or
- been explicitly marked as incompatible, redundant, or alternative

## Correctness and repair

If a candidate is faster but fails correctness:

- do not automatically discard it without evaluation
- classify the mismatch as:
  - trivial and likely fixable
  - promising but moderately costly to repair
  - fundamental and not worth pursuing

If the candidate is materially promising and likely repairable:

- open a bounded repair hypothesis or repair lane
- try to preserve the performance gain while restoring semantics

If the mismatch is not worth fixing:

- record it as a promising dead end and move on

## Closure

Hard closure may happen when:

- total iteration budget is exhausted
- the environment becomes unusable
- the benchmark or correctness workflow is no longer operable

Soft closure requires campaign review.

When closing the campaign:

- summarize best kept result
- summarize validated lanes
- summarize exhausted or invalidated lanes
- summarize promising but unmerged or unfixable ideas
- summarize why continuation is no longer justified
- leave durable state so a future agent can resume intelligently

## Durable memory obligations

Persist durable summaries of:

- baseline dossier
- current best candidate and why it is best
- lane list and statuses
- tested hypotheses and outcomes
- key evidence learned from failures
- integration opportunities
- repair-worthy incorrect candidates
- stagnation / exhaustion review conclusions
- closure reasoning
