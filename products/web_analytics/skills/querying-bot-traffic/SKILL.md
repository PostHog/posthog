# Querying bot traffic

PostHog classifies every event server-side using the visitor's user agent string. The classifier
sets three virtual properties that are queryable like any other event property — there is no
separate bot table. The classifier source of truth is
`posthog/hogql_queries/web_analytics/bot_definitions.py` (currently ~85 patterns), and the same
list is mirrored to the livestream service via `livestream/bot/definitions.json` (regenerate with
`hogli run build:bot-definitions`).

> **Detection scope.** Today this layer is purely user-agent matching. We're planning to add other
> identification techniques (e.g. IP/ASN signals, behavior heuristics) — when that lands, the
> virtual properties keep the same shape, so HogQL written against them today keeps working.

The virtual properties below are computed from the user agent at query time, so the same HogQL
works anywhere you have access to event properties — Web Analytics tiles, the SQL editor, insight
series, alerts. If you have a raw user agent string (e.g. from a custom event property or a data
warehouse table) and want to classify it ad hoc, the underlying functions are also exposed in
HogQL — currently behind a `__preview_` prefix while the API stabilizes:

- `__preview_isBot(user_agent)` → `Boolean`
- `__preview_getBotName(user_agent)` → matched bot display name, or empty
- `__preview_getBotType(user_agent)` → category (`ai_crawler`, `search_crawler`, …)
- `__preview_getBotOperator(user_agent)` → operator name (Google, OpenAI, …)

Use these when you need bot classification on something other than the standard `events` table —
they reuse the exact same `BOT_DEFINITIONS` source of truth as the virtual properties.

## Virtual properties on every event

| Property                 | Type      | Values                                                                                                                                                                            |
| ------------------------ | --------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `$virt_is_bot`           | `Boolean` | `true` if the event's user agent matched any pattern in `BOT_DEFINITIONS`; `false` otherwise                                                                                      |
| `$virt_bot_name`         | `String`  | Display name of the matched bot — e.g. `Googlebot`, `GPTBot`, `Claude`, `ChatGPT`, `Bingbot`, `ByteDance`. Empty string when `$virt_is_bot` is false.                             |
| `$virt_traffic_category` | `String`  | One of: `ai_crawler`, `ai_search`, `ai_assistant`, `search_crawler`, `seo_crawler`, `social_crawler`, `monitoring`, `http_client`, `headless_browser`. Empty for non-bot traffic. |

These are always present. Do not query `posthog:read-data-schema` to "look them up" — the schema
endpoint hides virtual properties.

## Event-type filter (important)

The bot dashboards filter to **only these five event types** when counting bot traffic:
`$pageview`, `$pageleave`, `$screen`, `$http_log`, `$autocapture`.

This matches the surface area where user-agent–based detection is reliable. Server-side events
without a real user agent (`event = '$identify'`, custom backend events) will have
`$virt_is_bot = false` even when generated by automation. Apply the same filter in any "bot count"
or "% of total" query so numerator and denominator stay comparable.

## Categories cheat sheet

| Category           | What it captures                                                | Examples                                                |
| ------------------ | --------------------------------------------------------------- | ------------------------------------------------------- |
| `ai_crawler`       | Bots that fetch pages to train or update an LLM                 | GPTBot, ClaudeBot, Google-Extended, Bytespider, Meta AI |
| `ai_search`        | Bots that fetch pages to answer a user's live AI search query   | OpenAI Search, Perplexity, Applebot                     |
| `ai_assistant`     | Browsing on behalf of an end user via an AI assistant           | ChatGPT-User, Claude-User, Perplexity-User              |
| `search_crawler`   | Traditional search engine indexers                              | Googlebot, Bingbot, Baidu, Yandex                       |
| `seo_crawler`      | SEO tools that catalog backlinks and rankings                   | Ahrefs, Semrush, Moz, Majestic                          |
| `social_crawler`   | Link unfurling for social platforms                             | Facebook, Twitter, Slack, LinkedIn, Iframely            |
| `monitoring`       | Uptime / performance probes                                     | Pingdom, UptimeRobot, Datadog                           |
| `http_client`      | Generic HTTP libraries (likely scripts or backend integrations) | curl, Python Requests, Go HTTP, Axios                   |
| `headless_browser` | Headless browsers (often QA, scraping, or AI agents)            | Headless Chrome                                         |

## Workflow

### 1. Confirm bot data exists

```sql
SELECT count() AS bot_events
FROM events
WHERE timestamp > now() - INTERVAL 7 DAY
  AND `$virt_is_bot` = true
  AND `$virt_bot_name` != ''
  AND event IN ('$pageview', '$pageleave', '$screen', '$http_log', '$autocapture')
```

If the count is zero on a high-traffic site, it likely means none of the captured events have a
real user agent (e.g., backend-only event capture).

### 2. Top bots by category

```sql
SELECT
    `$virt_bot_name` AS bot,
    `$virt_traffic_category` AS category,
    count() AS hits
FROM events
WHERE timestamp > now() - INTERVAL 7 DAY
  AND `$virt_is_bot` = true
  AND `$virt_bot_name` != ''
  AND event IN ('$pageview', '$pageleave', '$screen', '$http_log', '$autocapture')
GROUP BY bot, category
ORDER BY hits DESC
LIMIT 20
```

### 3. AI-only traffic over time

When the user asks about "AI bots", "LLM scrapers", or "AI traffic", combine the three AI
categories:

```sql
SELECT
    toStartOfHour(timestamp) AS bucket,
    `$virt_traffic_category` AS category,
    count() AS hits
FROM events
WHERE timestamp > now() - INTERVAL 24 HOUR
  AND `$virt_is_bot` = true
  AND `$virt_traffic_category` IN ('ai_crawler', 'ai_search', 'ai_assistant')
  AND event IN ('$pageview', '$pageleave', '$screen', '$http_log', '$autocapture')
GROUP BY bucket, category
ORDER BY bucket, category
```

### 4. % of total bot traffic

The numerator and denominator must use the **same event-type filter**, otherwise the percentage
is meaningless (this was a real bug in the bot tile — see PR #57062).

```sql
SELECT
    sumIf(1, `$virt_is_bot` = true AND `$virt_bot_name` != '') AS bot_events,
    count() AS total_events,
    100.0 * sumIf(1, `$virt_is_bot` = true AND `$virt_bot_name` != '') / count() AS bot_share_pct
FROM events
WHERE timestamp > now() - INTERVAL 7 DAY
  AND event IN ('$pageview', '$pageleave', '$screen', '$http_log', '$autocapture')
```

### 5. Most crawled paths

```sql
SELECT
    properties.$pathname AS path,
    count(DISTINCT `$virt_bot_name`) AS crawlers,
    count() AS requests
FROM events
WHERE timestamp > now() - INTERVAL 7 DAY
  AND `$virt_is_bot` = true
  AND `$virt_bot_name` != ''
  AND event IN ('$pageview', '$pageleave', '$screen', '$http_log', '$autocapture')
  AND properties.$pathname IS NOT NULL
GROUP BY path
ORDER BY requests DESC
LIMIT 50
```

### 6. Filtering bot traffic _out_ of regular analytics

Most product analytics queries should explicitly exclude bot traffic. Add a single guard:

```sql
WHERE event = '$pageview'
  AND `$virt_is_bot` = false
  AND timestamp > now() - INTERVAL 30 DAY
```

`$virt_is_bot = false` is preferable to `$virt_bot_name = ''` because it is set even when the
classifier matched a category but no name (rare but possible for new patterns).

## Filtering inside trends / funnels / other insight queries

Bot virtual properties are regular event properties from the insight schema's perspective. Add a
property filter with `type: "event"`:

```json
{
  "key": "$virt_traffic_category",
  "value": ["ai_crawler", "ai_search", "ai_assistant"],
  "operator": "exact",
  "type": "event"
}
```

For a trend showing AI-bot pageviews per day:

```json
{
  "kind": "TrendsQuery",
  "series": [
    {
      "kind": "EventsNode",
      "event": "$pageview",
      "math": "total",
      "properties": [
        { "key": "$virt_is_bot", "value": ["true"], "operator": "exact", "type": "event" },
        {
          "key": "$virt_traffic_category",
          "value": ["ai_crawler", "ai_search", "ai_assistant"],
          "operator": "exact",
          "type": "event"
        }
      ]
    }
  ],
  "interval": "day"
}
```

To invert it (humans only) flip `$virt_is_bot` to `["false"]` and drop the category filter.

## Tips

- **Always include the event-type filter** (`event IN ('$pageview', '$pageleave', '$screen', '$http_log', '$autocapture')`) when counting bot traffic — the bot tile uses it, percentages depend on it, and other event types don't carry meaningful user-agent classification.
- **Always set a `timestamp` filter** — `events` is high volume.
- The `$virt_*` properties are computed at query time from the user agent. Reclassifying a bot in a future PR will retroactively apply to historical events. There is no backfill needed.
- The full canonical bot list lives in `posthog/hogql_queries/web_analytics/bot_definitions.py`. To see what `$virt_bot_name` values you can expect, read that file. The TypeScript mirror is `frontend/src/lib/utils/botDetection.ts`.
- Empty strings, not nulls, are returned for the virtual properties on non-bot events. Filter with `!=  ''` rather than `IS NOT NULL`.
- Internal PostHog users can see live sample data for posthog.com bot traffic on the Lucas AI Farm project (id 371937) under Web Analytics → Bots.
