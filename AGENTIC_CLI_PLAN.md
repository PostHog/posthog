# Agentic CLI — Implementation Plan

> Rebuild the PostHog CLI so coding agents can reach every product surface the MCP exposes,
> via composable bash commands, generated from the same OpenAPI → MCP pipeline.

## Goal

Give coding agents CLI access to the ~388 product operations currently exposed only through
the MCP server, so they can **compose** operations with pipes/jq/scripts, **spill large data
to disk** instead of the context window, and **pay near-zero context tax** at session start
(no upfront tool definitions).

The bet (currently unproven): for agent workflows, a CLI beats MCP on **token usage**,
**task success rate**, and **time-to-completion**. Deliverable D2 exists to prove or kill that bet.

Three deliverables, from the source doc:

1. **D1 — The CLI** (autogenerated from the MCP pipeline).
2. **D2 — Proof of usefulness** (benchmarks vs MCP).
3. **D3 — Blog post** (gated on D2 results).

## Approach (named): "Rust-native runtime over a fully-declarative manifest, with a CI conformance gate"

- **Codegen stays in TypeScript**, inside the existing MCP pipeline (`services/mcp/scripts/generate-tools.ts`).
  We extend it to emit one committed artifact, `cli-manifest.json` — the _single source of truth_ for
  the CLI's command surface. This is the prior PR's best idea, kept.
- **The manifest is 100% declarative.** Every request-shaping transform the MCP performs is encoded as
  data (param buckets, casts, renames, `soft_delete`, `inject_body`, `query_kind`, query schemas,
  actors source→endpoint map, filter-nesting rule, response-filter patterns). No imperative tool logic
  lives only in code.
- **The runtime is Rust**, implemented as an extension of the existing `cli/` crate. It is a _generic
  manifest interpreter_ with a small fixed set of transform primitives. New tools / params / query kinds
  flow through with **zero Rust changes** — only a brand-new _category_ of transform would need code.
- **Behavior parity is a CI invariant**, enforced by a conformance harness that compares the Rust
  request-builder's output to the MCP handler's output for a corpus of cases.
- **Command shape is hybrid** (your choice): scalar top-level params become real `--flags` with native
  `--help`; nested/complex params go through `--json`. Discovery via `list` / `schema` verbs _and_
  per-command `--help`.

### Why extend the existing Rust CLI instead of a new binary

The existing `cli/` already implements `login`, credential storage, `InvocationContext`, the
`PHClient` (auth header, `--host`, rate limiting, SSL), structured error capture, and self-telemetry.
Extending it means the new commands inherit all of that. The prior PR's entire P1 bug class
(credential override, env-var bridge, node-path resolution) existed _solely_ because it execed a Node
process from Rust. Staying in-process deletes that class entirely. We also avoid shipping/maintaining
two CLIs.

The local-processing commands the current CLI already has (`sourcemap`, `dsym`, `hermes`, `proguard`,
`symbol-sets`) have **no OpenAPI equivalent** — they do local file work. They stay exactly as they are.
The generated API surface is additive.

## Alternatives considered

| Option                                                            | What it is                                                                                  | Why rejected (or kept as fallback)                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------------------------------------------------------------- | ------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Standalone TS CLI via `npx`** (Cloudflare model)                | New Node CLI in `services/`, reuses MCP transform functions directly, always-latest via npx | Zero divergence by construction _if it imports MCP's own functions_ — genuinely attractive. Rejected for v1 because: (a) **Node cold-start tax per invocation** is a real cost when an agent runs dozens of commands in a loop; (b) it creates a **second CLI** to distribute and document alongside the Rust one; (c) it forfeits the free auth/telemetry/rate-limit inheritance. Kept on the table as the body of the **hybrid fallback**. |
| **Hybrid Rust→Node exec** (the prior PR, #53345)                  | Rust shim execs a bundled Node CLI, injects creds as env vars                               | Rejected. Inherits the credential-bridge P1 bugs, fragile node-path resolution, and a hidden Node dependency inside a "Rust" binary. The worst structural decision in the prior attempt.                                                                                                                                                                                                                                                     |
| **Rust-native runtime + declarative manifest + conformance gate** | This plan                                                                                   | **Chosen.** Best runtime UX (instant startup, single static binary, matches existing distribution), inherits auth/telemetry, and the parity risk is bounded to 2 transforms + neutralized by the conformance harness.                                                                                                                                                                                                                        |
| **Pure flag-per-parameter command tree**                          | `ph feature-flag create --key x --name y`, full clap tree                                   | Rejected as the _only_ model: 388 tools × many params is an enormous generated surface, and nested query/property bodies don't map to flat flags. Folded into the **hybrid** shape you chose (scalars as flags, nested as `--json`).                                                                                                                                                                                                         |

## Key technical decisions

### 1. Manifest is the single source of truth, generated by the MCP pipeline

- **What:** Add a `generateCliManifest()` step to `generate-tools.ts` that runs off the _already-resolved_
  YAML + OpenAPI inputs (same skips, overrides, descriptions, scopes, annotations the MCP handlers use).
  Commit `cli-manifest.json`.
- **Why:** One pipeline, one source of truth, zero duplicate handler code. The CLI inherits every future
  product change automatically the same way the MCP does.
- **Tradeoff:** The manifest must be regenerated and committed when serializers/YAML change — same
  discipline already required for `api.schemas.ts` etc. CI must verify it's not stale (a `--check` mode).

### 2. The manifest must be _fully declarative_ — close the 2 gaps the prior PR left open

The audit found 15/17 transforms already declarative. The two that aren't must be encoded as data:

- **Query-wrapper wrap (Transform 9):** embed `query_kind`, the flattened `query_schema`, referenced
  `types`, and a declarative `filter_nesting` rule for the `filterGroup`→`PropertyGroupFilter` transform.
- **ActorsQuery routing (Transform 10):** embed a `actors_source_map` (`source.kind` → backend method/endpoint)
  so the runtime routes `query-*-actors` exactly like MCP, instead of naively POSTing to `/query/`
  (the prior PR's silent bug). If `query.source.kind` routing can't be fully captured declaratively for
  some future actors variant, that specific tool is excluded from the CLI until it can — never shipped divergent.
- **Tradeoff:** A richer manifest schema. Worth it — it's the thing that makes Rust-native safe and keeps
  _any_ runtime honest.

### 3. Runtime = generic interpreter with fixed transform primitives

- **What:** Rust code implements a closed set of primitives: path-substitution (always `encodeURIComponent`),
  query-string serialization (skip null/undefined/empty-array, JSON-stringify objects — mirror
  `api/client.ts`), body collection/whitelist, casts (`string-int`), renames, `inject_body`,
  `soft_delete`→PATCH, query-wrap, actors-route, response-filter. The manifest selects which apply per tool.
- **Why:** New tools never touch Rust. Only a _new kind_ of transform (rare) does.
- **Tradeoff:** This is a parallel implementation of transforms 2/3/4/etc. that must track the MCP's. Bounded,
  and guarded by the conformance harness (Decision 4). The maintenance tax is "new transform category,"
  not "new tool" — infrequent.

### 4. Behavior parity is enforced in CI (the load-bearing guardrail)

- **What:** A conformance harness. A corpus of `(tool, params)` cases (hand-authored for the hard ones,
  fuzzed/generated for breadth) is run through (a) the MCP handler's request-builder and (b) the Rust
  request-builder. Assert **identical method + path + query + body**. Run in CI on every change to the
  manifest, the generator, or the Rust runtime.
- **Why:** Makes "the CLI does not change tool behavior" a _verifiable invariant_, not a promise. Catches
  drift the moment the query schema or a transform evolves. This is what lets us choose Rust-native at all.
- **Mechanics:** Easiest seam is to have the MCP test build emit golden request JSON per corpus case;
  the Rust test reads the goldens and asserts equality. The MCP side already has the handlers; we just
  capture the request instead of sending it (dry-run mode).
- **Scope of the contract:** parity covers the **request** (the side-effecting part). **Response
  presentation** (enrichment, formatted output) is explicitly _out_ of the parity contract — see Decision 6.

### 5. Command shape: per-category commands, hybrid flags + `--json`

- **What:** Commands are structured **per category**: `posthog-cli <category> <verb> [flags] [--json '{...}']`
  (e.g. `posthog-cli feature-flag create --key x`). The `clap` command tree is built **dynamically at
  runtime from the manifest** (builder API). Per param, the manifest marks `flag_eligible` (scalar,
  top-level → real `--flag`) vs json-only (nested object/array → carried in `--json`). Precedence:
  explicit flags override keys in `--json`.
- **Why per-category (not an `api <tool>` namespace):** matches the CLI idioms agents are trained on
  (`git commit`, `gh pr create`, `docker run`) far better than a flat namespace. It also yields **native
  `clap --help` discovery for free** at every level (`posthog-cli feature-flag --help` lists the verbs;
  `... create --help` lists the flags) — Georgiy's `--help` proposal, no custom code.
- **Cost we accept:** tool names today are irregular (`create-feature-flag`, `feature_flags_list`, …), so the
  generator must **derive a deterministic `<category> <verb>`** for each tool, and we lose the 1:1 mapping
  to MCP tool names. _Mitigation:_ (a) a single documented derivation rule (category from the manifest's
  `category`/`feature`; verb from the operation); (b) keep the original MCP tool name as a discoverable
  **alias** in `list`/`schema` output and `--help`, so knowledge transfers from the MCP. Also need a rule
  for collisions between generated category names and the hand-written local commands (`login`, `sourcemap`,
  `dsym`, `hermes`, `proguard`, `symbol-sets`, `query`).
- **Discovery (two tiers):** native `--help` for humans; plus machine-readable `posthog-cli list [--category]
[--json]` and `posthog-cli schema <category> <verb> [--full]` for token-efficient agent discovery
  (brief by default, `--full` adds descriptions).
- **Tradeoff:** Two ways to pass a value = more codegen and a small risk of agent confusion. Mitigated by a
  clear precedence rule and `--help` text that states which params are flag-eligible.

### 6. Output: raw JSON by default, opt-in `--enrich` _(decided)_

- **What:** `exec` prints raw API JSON to stdout (pipes to `jq`). `--fields a,b,c` masks output (preserving
  `next`/`previous`/`count`). The MCP's `_posthogUrl` enrichment and `formatted_results` presentation are
  **not** applied by default; `--enrich` opts into MCP-identical responses.
- **Why:** The CLI's whole value proposition is composability — presentation is the agent's job, and raw
  JSON pipes cleanly. Enrichment is a UI affordance for chat clients.
- **Parity note:** the conformance contract covers **requests** (the side-effecting part); response
  presentation is deliberately outside it. `--enrich` exists for callers that want byte-parity responses.

### 7. Validation happens client-side, before the request

- **What:** `clap` enforces required flags and types natively; `--json` is validated against the manifest's
  param types/enums/required before sending. Casts (`string-int`) applied client-side.
- **Why:** Fast feedback loop for agents (the doc's Question #1) — reject a malformed call locally instead
  of round-tripping to a 400. This is the CLI answer to "MCP exposes a schema you can validate against."
- **Tradeoff:** The manifest must carry enough type info (it already does for query schemas; extend to REST
  params). We won't replicate _full_ server-side validation — just the cheap, high-value checks.

### 8. Distribution: embed the manifest in the binary; upgrade via normal CLI version bump

- **What:** `include_str!("cli-manifest.json")` at build time → discovery and execution need zero network.
  The manifest is version-locked to the binary. Upgrades follow the existing cargo-dist / npm `cli`
  package / brew flow. Agents are steered via a one-line `AGENTS.md` injection (the doc's discovery answer).
- **Why:** Instant startup, offline discovery, no version-skew between binary and manifest. Avoids the
  `npx`-always-latest startup tax.
- **Tradeoff:** A stale local binary has a stale tool surface until upgraded. Acceptable — same as any CLI.
  (Optional later: a `posthog-cli api refresh` that fetches a newer manifest the running binary can
  interpret, _only_ if the manifest schema version is compatible.)

### 9. Skills are distributed through the CLI, opt-in _(decided)_

- **What:** `posthog-cli skill list` and `posthog-cli skill install <name>` (top-level `skill` group, for
  consistency with dropping the `api` namespace in Decision 5). The CLI is the _delivery mechanism_, but
  installation is **explicit/opt-in** — never auto-installed (Browserbase model; skills evolve faster than
  the CLI and carry workflow intent).
- **Why:** Keeps the binary tool-agnostic while still making skills one command away, and lets skills ship on
  their own cadence independent of CLI releases.
- **Residual sub-decisions:** (a) **source of truth** for the skill registry — reuse the existing skills-store
  the MCP already references, fetched at runtime (so it's not version-locked to the binary like the tool
  manifest is); (b) **install target dir** and format coupling — skills are an agent-specific format
  (`SKILL.md`), so `install` needs a target convention (e.g. `.claude/skills/` / `.agents/skills/`) and
  should state the format it writes. Steer agents to the `skill` commands via the `AGENTS.md` line.

### 10. Project-scoped by default; mirror the existing CLI _(decided)_

- **What:** Default scope is the project, resolved exactly like the current CLI (`POSTHOG_CLI_PROJECT_ID` /
  stored login / `--host`), reusing `InvocationContext`. `{project_id}` is auto-filled from context.
- **Org-scoped tools:** `{organization_id}` tools resolve the org from the active project's org when that's
  derivable, else require an explicit `--organization-id`. Unlike the prior PR, org params must appear in
  `schema`/`--help` output (the prior PR hid them, which is **O5**'s failure mode — now closed).

## Implementation outline

### D1 — The CLI

**Phase 0 — Spike & the Go/No-Go gate (smallest possible, highest-risk-first)**

- Pick the _worst_ cases on purpose: one REST tool (e.g. `feature-flag` CRUD), one query wrapper
  (`query-trends`), and one actors tool (`query-trends-actors`).
- Extend `generate-tools.ts` to emit a declarative manifest for _just these_, including the query-wrap +
  filter-nesting + actors-source-map.
- Build a minimal Rust interpreter for them and stand up the conformance harness on these cases.
- **GATE:** Does the Rust request-builder produce byte-identical requests to MCP for all spike cases,
  including the actors routing? **Yes →** proceed Rust-native. **No, and a transform proves irreducibly
  dynamic →** fall back to the standalone-TS runtime (import MCP functions directly) for that subset or
  wholesale; the manifest/codegen/UX work below is unaffected by that swap.

**Phase 1 — Manifest + interpreter + parity harness (the foundation)**

- Generalize the manifest schema to cover all 17 transforms; document the schema.
- Implement the full generic Rust interpreter (all transform primitives).
- Build out the conformance corpus (hand-authored hard cases + generated breadth) and wire it into CI,
  plus a manifest-staleness `--check` in CI.

**Phase 2 — Command UX**

- Dynamic per-category clap tree from the manifest; hybrid flags + `--json`; precedence rule; the
  `<category> <verb>` derivation + local-command collision rules.
- `list [--category] [--json]` / `schema <category> <verb> [--full]` machine discovery; native `--help`.
- `skill list` / `skill install` (Decision 9).
- Client-side validation; `--dry-run`; `--fields`; raw-JSON output; `--enrich` opt-in.

**Phase 3 — Scale to all ~388 tools + hardening**

- Turn on the full surface. Expand the conformance corpus to sample every transform category and every
  domain.
- **Security pass (explicit):** path-param `encodeURIComponent` everywhere (the prior PR's injection bug),
  scope-aware error messages, and — because agents act on **untrusted input** (issue bodies, PR comments) —
  ensure no command path is shell-evaluated and that path params can't redirect to a different endpoint.
- Auth: confirm reuse of `login`/credentials/scopes; per-command required-scope hints in `--help`.

**Phase 4 — Distribution & steering**

- `include_str!` the manifest; cut a release through the existing pipeline.
- Add the `AGENTS.md` steering line. Update `cli/README.md` and `docs/`.

### D2 — Proof of usefulness (evals vs MCP)

**Reuse the existing eval harness — do not build a bespoke benchmark.** PostHog already has a
Braintrust-backed eval framework at `ee/hogai/eval`, including a **sandboxed** harness
(`ee/hogai/eval/sandboxed/`) that runs a real coding agent in a Docker sandbox against a live demo
project, and parses the agent's tool/Bash calls with deterministic + LLM-judge scorers. Critically,
it already has a `cli_mcp/` suite — but that exercises the **MCP's `exec` (CLI-like) tool mode**, not
our actual `posthog-cli` binary. So we don't yet have evals for the binary; we add them.

- **New suite:** `ee/hogai/eval/sandboxed/cli_posthog/` mirroring `cli_mcp/`. The sandbox gets the
  `posthog-cli` binary on PATH + the steering block (via `posthog-cli init`); the agent drives it
  through `Bash`, not MCP tool calls.
- **Reuse:** the `SandboxedPublicEval` harness, demo-data setup, Braintrust reporting, and generic
  scorers (`ExitCodeZero`, post-processing checks). **New:** Bash-aware scorers that parse
  `posthog-cli <category> <verb>` invocations (did it discover via `--help`? use `--dry-run` before a
  write? pick the right command + flags?), paralleling `cli_mcp/scorers.py`.
- **Comparative arm:** run the same task suite against the MCP arm (the existing `cli_mcp`/tools-mode
  suites) to get the head-to-head the thesis needs: tokens, success rate, time-to-completion.
- **Tasks:** realistic, composition-heavy multi-step tasks (e.g. "find the flag gating X and roll it
  back", "pull last week's signups bucketed by source"), plus some single-call tasks so we don't
  cherry-pick. Cases are defined like the existing `SandboxedEvalCase` (name, prompt, expected dict).
- **Prereqs:** the sandbox image must include the built binary; the agent must be steered to it
  (`steering.md` / `init`). These are the only genuinely new infra pieces.
- **Gate:** D3 (blog) only ships if results support the thesis. If they don't, that's worth knowing
  before committing the org to the CLI direction.

This doubles as the project's eval coverage (the "do we have evals?" question): the `cli_posthog`
suite is both the proof-of-usefulness benchmark and the ongoing regression eval for the CLI.

### D3 — Blog post

- Narrative (why agents love CLIs, the composition/token argument), the architecture (manifest-from-MCP,
  parity harness), and the **measured** results from D2. Public-OSS-safe (no internal scale/incident data).

## Risks & failure modes

- **Silent behavior drift** (the #1 risk you raised). _Mitigation:_ the conformance harness in CI is the
  whole point — drift fails the build. Without it, do not ship Rust-native.
- **A query transform proves irreducibly dynamic.** _Mitigation:_ Phase 0 gate; standalone-TS fallback for
  the affected subset; or exclude that tool until declarativizable.
- **Manifest staleness** (manifest ≠ current serializers). _Mitigation:_ CI `--check` that regenerates and
  diffs, same as other generated artifacts.
- **Untrusted-input exploitation** — agents run CLI commands derived from issue/PR text. _Mitigation:_
  path-param encoding, manifest-bounded args, no shell-eval, scope enforcement server-side anyway.
- **The thesis doesn't hold** (CLI not actually better than MCP). _Mitigation:_ that's exactly what D2
  measures; we find out cheaply before the blog and before deprecating anything.
- **Maintenance tax of a parallel Rust transform impl.** _Mitigation:_ bounded to new transform categories
  (rare); conformance harness catches anything missed.
- **Dynamic clap complexity.** Low — clap's builder API supports runtime-constructed command trees; the
  spike de-risks it.

## Decisions log (O1–O5 resolved)

- **O1 — Response parity:** _Resolved._ Raw JSON by default, `--enrich` opt-in. See Decision 6.
- **O2 — Skills:** _Resolved._ Distributed via the CLI, opt-in (`skill list` / `skill install`). See Decision 9.
  Residual: skill registry source + install target dir/format.
- **O3 — Non-bash clients / MCP-proxy:** _Resolved — out of scope._ Not because it's a big lift (a CLI-shelling
  proxy is small-to-medium), but because the **existing native MCP already serves every non-bash client**. A
  proxy only matters if we later sunset the MCP — a strategic call for _after_ D2, not a v1 task.
- **O4 — Command structure:** _Resolved._ Per-category (`posthog-cli <category> <verb>`), no `api` namespace.
  See Decision 5. Residual: the `<category> <verb>` derivation rule + collision rule with local commands.
- **O5 — Scope default:** _Resolved._ Project-scoped, mirrors the existing CLI; org tools surface explicitly.
  See Decision 10.

## Remaining residual decisions (smaller, can settle during Phase 0/1)

- Skill registry source of truth + install target dir/format (from O2).
- The deterministic `<category> <verb>` naming rule and the generated-vs-local-command collision rule (from O4).
- Org resolution rule for `{organization_id}` tools (from O5).

## Out of scope (v1)

- Replacing/rewriting the local-processing commands (`sourcemap`, `dsym`, `hermes`, `proguard`,
  `symbol-sets`) — they have no API equivalent and stay as-is.
- An MCP-proxy for non-bash clients (O3) — redundant while the native MCP exists.
- Full server-side-equivalent client validation (we do cheap, high-value checks only).
- Deprecating the MCP server — the CLI is additive; any MCP sunset is a separate decision informed by D2.
